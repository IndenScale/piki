软件定义工程（SDE）：大模型落地工业的必由之路

当前的大模型是"盲"的，当前的工业软件是"给健全人类用的"。当盲人试图指挥健全人操作复杂机器，摩擦与失败不可避免。解决之道不是让大模型视力更好，而是让工业软件变成可被非人类智能体直接操作的基础设施。

一、一场正在发生的错配

2024 年起，工业界对大模型的热情从"聊天"迅速转向"落地"。最常见的落地思路是：把大模型接到现有工业软件上，让它像人一样看界面、点按钮、填表单、出图纸。

这个思路很快遇到天花板：

  • LLM 是盲的。它看不到屏幕，无法确认一个设备是否真的放在了正确的 U 位，无法判断线缆是否真的接对了端口。
  • VLM 是散光的。它能截图、能识别 UI 元素，但定位不精确、对长程依赖理解差、对工程语义几乎无知。它可以把"插入光纤"理解为点击某个坐标，但无法验证这根光纤的速率、极性、长度是否匹配。
  • 通过 UI 操作慢、不准确、难交验。每个动作都要经过屏幕渲染、元素检测、坐标点击、结果观察，延迟高、鲁棒差、结果难以被其他系统复用。

与此同时，工业软件本身并没有为被机器操作而设计。它们的默认假设是：使用者视力良好，受过专业训练，熟悉鼠标、菜单、对话框和快捷键。这个假设在过去三十年是合理的，但在 Agent 时代变成了根本性的障碍。

这不是技术细节问题，而是交互范式的不匹配：一边是试图理解世界的多模态大模型，一边是为人类眼手协调优化的 GUI。让前者驱动后者，就像让一位盲人通过语音指挥一位健全人操作复杂机床——不是不可能，但成本高、错误多、难以规模化。

二、两个被忽视的结构性事实

事实一：大模型的"中期训练"正在代码领域发生

当前工业界讨论大模型时，经常强调"工业数据稀缺"。这个判断是对的，但只说对了一半。

更关键的事实是：大模型的中期训练正在以 Code RLVR 为主。RLVR（基于可验证奖励的强化学习）已经在数学、代码等领域证明了规模化提升模型能力的路径。其前提是：存在大量可验证、可自动评分的任务。

代码之所以成为 RLVR 的主战场，不是因为代码数据多，而是因为代码拥有从语法到运行时反馈的完整可验证链条：

验证层级 代码领域 工程设计领域
L0 格式 语法高亮、缩进 文件格式合法性
L1 Schema 类型系统 字段约束与值域
L2 完整性 单元测试 引用完整性、接口兼容性
L3 业务规则 集成测试 功率预算、空间冲突
L4 运行时 端到端测试 几何碰撞、物理干涉
L5 仿真 性能基准 热、流体、电磁仿真
L6 评估 人工代码评审 AI/专家综合评估

工程设计天然具备类似的结构：约束、规则、物理定律、制造规范、验收标准。但工业软件没有把这些结构暴露给模型。它们把工程语义埋在菜单、对话框、私有文件格式和云端数据库里。模型无法直接读取设计意图，无法直接获得可验证的奖励信号，因此无法在工程领域建立有效的 RLVR 飞轮。

工业数据稀缺是结果，工业软件不可被机器操作才是原因。

事实二：工业软件假设使用者是"视觉与操作能力正常的人类"

这是绝大多数工业软件在设计时隐含的假设：

假设 对 Agent 的影响
用户能通过视觉快速理解复杂界面 Agent 需要 VLM + OCR + 元素检测，成本高且易错
用户能根据经验判断哪些菜单里有需要的功能 Agent 需要大量领域特定的 UI 探索策略
用户能接受"在对话框里逐步输入" Agent 必须模拟鼠标键盘，动作序列冗长
用户会主动保存、命名、管理文件 Agent 需要额外的状态追踪和恢复机制
用户能理解专有文件格式和二进制数据 Agent 无法直接读取、diff、验证设计意图

这些假设在 GUI 时代是优势，在 Agent 时代是负债。它们让工业软件成为一座设计精美但只对人类开放的建筑——入口狭窄、台阶陡峭、没有盲道、没有电梯。

三、工业软件无障碍化：Headless Engineering

面对这个错配，最常见的补救思路是给工业软件加 API 包装层——用 RPA、UI 自动化、插件系统让 Agent 间接操作旧软件。

这实际上是在"盲人指挥健全人"的模型上打补丁。它有几个致命缺陷：

  • 语义层仍然是 GUI。API 包装层传递的是 UI 事件（点击、输入、拖拽），而不是设计意图（"SRV-01 安装在 RACK-A01 的 U18-19，功率分配来自 PDU-A 端口 3"）。
  • 验证代价昂贵。每次操作后要截图确认、OCR 识别、状态推断——成本高、可靠性差。
  • 不可组合。多个 Agent 不能同时操作一个界面，两个设计变更不能通过 git merge 自动合并。
  • 不可版本化。旧软件里的"版本历史"是操作序列的录像，不是设计意图的 diff。

正确的路径是工业软件无障碍化——不是给旧软件加辅助功能，而是重新设计底层，使设计意图以可被机器直接操作的形式存在。我们称之为 Headless Engineering：

  1. 设计意图以文本声明形式存在。用人类可读、机器可解析的声明式语言描述"什么东西存在、放在哪里、怎么连接"，而不是用点击和拖拽。
  2. 规则引擎提供秒级、确定性、可解释的验证反馈。Agent 提交修改后，几百毫秒内就能知道是否符合约束——不是一个需要人类用眼睛确认的可视化结果，而是一组结构化的 pass/fail 诊断。
  3. 生成器将文本声明转化为交付物。BOM 清单、配电图、施工图、电缆表等交付物不再是人工绘制的结果，而是从文本声明出发的可重复生成产物。
  4. 可视化是消费层，不是设计层。3D 预览、AR 叠加、图纸等人类需要的视觉输出仍然存在，但它们是只读的——看到问题后，修改必须回到文本或 Agent。

    5. 四、SDE 的核心类比：工程设计 = 软件工程

    在 SDE 的框架下，一次设计迭代看起来像软件工程：

    
写设计声明 → 规则检查（多层自动校验） → 修正违规 → 生成交付物 → git commit

    这不是隐喻，而是真实的 workflow。工程师（或 Agent）用声明式语言定义设备、布局、连接和配合。规则引擎在秒级内从语法到空间碰撞逐层检查。诊断信息精确到行号，可以逐条修复。交付物从声明自动生成，而非人工绘制。文本文件天然支持 diff、review、merge、blame。

    错误在提交前被发现，而不是在现场施工后。

    这就是 SDE 的本质：不是用软件辅助人来设计工程，而是让工程设计本身像软件一样被定义、被验证、被版本化、被自动化。

    五、SDE 的未来蓝图

    1. 规则市场：从项目经验到行业知识

    • 国标、行标以开源规则包形式维护
    • 设备厂商发布官方型号数据与配套验证规则
    • 设计院把项目特殊要求贡献回社区规则库
    • 学术界把研究成果沉淀为可复用的检查器

    2. 工程 Agent 的 RLVR 训练场

    • 模型提出设计方案（修改文本声明）
    • 系统在秒级返回规则检查反馈
    • 高成本仿真和专家评估为少数高质量样本提供最终奖励
    • 模型在数百万次自动迭代中学会多目标权衡、长程规划和物理一致性

    3. 跨企业、跨仓库的设计协作

    • 每个参与方维护自己的设计仓库
    • 通过标准引用机制互相引用资产与约束
    • 区分"我的设计范围"和"我引用的外部约束"
    • 接口变更通过 CI 自动触发依赖方规则检查

    4. 制造端的非确定性

    一个诚实的限定：SDE 的"编译"类比不能无限延伸。

    软件编译是确定性的——同样的源代码产出的字节码完全一致。工程设计的"编译器"却是供应链、非标物料、车间机床和现场施工人员。文本声明中一个完美的 rack-mount-19inch 配合约束，在现场可能因为钣金件 2mm 的加工形变而完全失效。公差、良率、材料批次差异——这些物理世界的噪声，是当前声明式文本难以完全建模的。

    这不是否定 SDE，而是划定了它的边界：SDE 保证的是设计意图的正确性，不是物理实现的完美性。前者是必要条件，后者需要制造端的标准化配合。

    这自然引出一个更大的命题——制造即服务（Manufacturing as a Service, MaaS）：当各个加工厂乃至加工机器人能够接收标准的工艺文件并确定性执行加工时，SDE 的"编译"类比才能真正闭环。从设计声明到工艺文件到加工执行，形成一条完整且可验证的单向数据流。这是 SDE 的下一个前沿。

    5. 从设计意图到制造安装的单向数据流

    • 工厂从结构化 BOM 直接读取采购数据
    • 数控机床从开放几何格式读取加工数据
    • 施工队从施工图和线缆表按图施工
    • 运维系统从设计仓库继承资产台账和连接关系

    六、为什么是现在

    SDE 之所以在这个时刻成立，是因为三个条件同时成熟：

    1. 大模型能力达到临界点：虽然不能可靠操作复杂 GUI，但已经能较好地读写结构化文本、解释规则、生成 diff。
    2. 工程领域对自动化的需求迫近极限：人力成本上升、项目复杂度上升、交付周期缩短，传统"人眼+鼠标"模式难以为继。
    3. 开放生态成为竞争优势：专有格式和封闭接口在 Agent 时代会加速贬值，能够被机器读写的开放系统将获得网络效应。

      4. 谁先完成 SDE 转型，谁就能让 AI 真正进入工程核心流程；谁固守 GUI-only 的旧范式，谁就会发现自己拥有世界上最精良的工具，却没有任何 Agent 能高效使用它。

      七、结语：不是替代工程师，而是升级工程

      SDE 不是要让 Agent 取代工程师。它的目标是：

      让工程师从"画图+点击+校对"的执行者，变成"定义意图和规则、审阅 Agent 输出"的设计架构师。

      人类仍然是不可替代的：复杂权衡、现场经验、价值判断、创造性设计。但这些高级能力只有在低级执行工作被有效自动化的前提下，才能得到最大程度的释放。

      大模型是盲的，工业软件是给人看的。这个错配不会通过让大模型"视力更好"来解决——那太慢、太贵、太脆弱。正确的路径是：让工业软件变得对机器友好，让工程设计变得像软件一样可定义、可校验、可协作。

      这就是 SDE。
