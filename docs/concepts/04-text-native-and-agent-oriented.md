# Text-Native & Agent-Oriented

> piki 不是“让人写 YAML 的工具”，而是“让 Agent 和人类都能可靠操作工程设计”的基础设施。

---

## 背景

工程设计的传统界面是 GUI：AutoCAD、Revit、BIM 工具、网管系统。工程师通过点击、拖拽、画图来表达设计意图。这种交互方式适合人类直觉，但带来了版本控制困难、规则难以自动化、AI 难以理解等问题。

piki 选择 Text-Native（以结构化文本为唯一真相源），并不是为了否定 GUI，而是为了让设计数据具备：

- 可版本控制（`git diff`）
- 可自动验证（Python 规则）
- 可被 AI 可靠读写
- 可跨工具组合

随着 AI Agent 的能力快速增强，Text-Native 的价值从“工程师 readable”升级为“Agent operable”。piki 的交互范式因此进一步定义为 **Agent-Oriented**：人类通过自然语言、草图、GUI 视图与 Agent 交互，Agent 在文本真相源上执行具体修改、校验与生成。

---

## 核心假设

以下假设指导 piki 的产品与架构决策。

### 1. 文本是机器协作的最小公分母

| 表示形式 | 人类可读 | 可 diff | 可程序化修改 | 可验证 | Agent 友好 |
|---|---|---|---|---|---|
| 手绘图纸 | ⚠️ | ❌ | ❌ | ❌ | ❌ |
| Excel | ⚠️ | ❌ | ⚠️ | ❌ | ⚠️ |
| CAD/BIM 二进制 | ❌ | ❌ | ❌ | ❌ | ❌ |
| 闭源 SaaS | ⚠️ | ❌ | 依赖 API | ⚠️ | ⚠️ |
| Text-Native YAML | ✅ | ✅ | ✅ | ✅ | ✅ |

GUI 是人类消费的友好界面，但**文本是机器协作的通用协议**。

### 2. 人类不会直接写 YAML，Agent 会

大多数终端用户——尤其是非 STEM 背景的设计师、产品经理、社交媒体创作者——不会手写结构化文本。他们的自然交互方式是：

- 自然语言描述需求
- 草图或参考图
- 点击“检查”“优化”“生成报告”

Agent 负责把这些高层意图翻译成 YAML 修改、规则运行、版本提交。因此，piki 的文本层不是为人类键盘输入设计的，而是为 **Agent 可编程性** 设计的。

### 3. GUI 会退化为可视化消费层

Studio、3D 预览、机柜面板图、线缆清单等 GUI 价值不会消失，但它们的角色从“设计编辑器”变为“设计监视器”：

- 展示当前文本真相源的状态
- 高亮规则 violations
- 支持人类审阅 Agent 提出的变更
- 在受控场景下（如草图导入）触发 YAML 生成

**设计修改的入口优先是 Agent，而不是鼠标。**

### 4. 增量用户没有 GUI 惯性，存量工程师会被竞争压力改造

新一代工作者已经习惯向 AI 询问作业、计划、情感建议。他们不会接受先学六个月 Revit 再开始设计。对“增量用户”而言，Agent 是第一界面。

存量专业工程师的 GUI 肌肉记忆很强，但随着 Agent 输出质量提升，坚持使用手动 GUI 的成本会越来越高。最终只有愿意成为“工作流架构师”的工程师会保留核心设计权，其余执行性工作被 Agent 替代。

piki 的策略是：**先服务愿意拥抱 Text-Native 的增量团队，再向存量市场渗透**。

## 对 piki 架构的 implications

这些假设直接转化为 piki 的技术边界。

### 1. 唯一真相源是文本

`instances/`、`models/`、`rules/` 下的 YAML 和 Python 文件是设计的唯一真相源。任何渲染、报告、预览都是派生物。

### 2. Studio 是可视化补充，不是替代品

Piki Studio 读取本地 YAML 构建场景，提供 3D 预览和属性面板。它不会直接修改设计文件；如果需要修改，应通过 Agent 或受控生成器产生 YAML diff，再供人类审阅。

参见 [ADR-009: Piki Studio](../adr/009-piki-studio.md)。

### 3. Agent 接口是第一优先级

piki 的演进方向是提供稳定的 Agent API，使外部 Agent 能够：

- 读取当前设计状态
- 提出符合 schema 的修改
- 运行规则检查并解释结果
- 生成报告和交付物

人类界面可以多样化，但 Agent 必须能可靠地操作文本层。

### 4. 规则库是核心护城河

Agent 的价值取决于它能执行多少领域规则。开源、可扩展的规则库让 piki 从“一个检查工具”进化为“工程领域的通用操作系统”。

---

## 边界与妥协

完全拒绝 GUI 会提高 adoption 门槛。以下场景允许 GUI 作为入口，但输出必须是文本：

| 场景 | 允许的人类输入 | 输出形式 |
|---|---|---|
| 草图导入 | 手绘/截图 | Agent 生成 YAML 初稿 |
| 自然语言设计 | 语音/文本 | Agent 生成 YAML diff |
| 可视化调整 | 拖拽、修改参数面板 | Agent 重写对应 YAML |
| 报告审阅 | 点击查看、批注 | 人类批注存入文本评论/规则覆盖 |

关键原则：**GUI 可以是输入方式，但永远不能绕过文本真相源。**

---

## 检验指标与推翻条件

这些假设不是教条。若出现以下信号，应重新审视：

- 多模态 Agent 在 GUI 上的操作精度、可复现性超过文本操作；
- 企业客户大规模拒绝文本层，即使 Agent 能完全代理；
- 主流 CAD/BIM 厂商开放出比 Text-Native 更适合 Agent 操作的中间格式。

在此之前，piki 坚持 **Text-Native & Agent-Oriented**。
