# 为什么需要 piki

> 工程设计中的错误，发现得越晚，代价越高。

## 一个问题

假设你是一名网络工程师，负责机房设备扩容。你设计了一个方案：新增 2 台交换机和 1 台防火墙，放在机柜 R-01 的 U20-U18 位置。

你检查了 U 位，没有冲突。你检查了线缆长度，都在合理范围内。你提交了方案，评审通过，施工完成。

上电后，PDU-A 跳闸了。

**为什么？**

因为你忘了算功率。PDU-A 上已经有 2 台核心交换机（每台 350W），新增 2 台接入交换机（45W + 65W），总负载 810W。看起来没问题——PDU-A 额定 3000W，810W 只有 27%。

但施工队按惯例给每台设备留了 20% 余量，实际按 TDP 的 1.2 倍计算。而且你没注意到，PDU-A 的某个插座接触不良，实际可用容量只有 2500W。

上电瞬间，所有设备同时启动，浪涌电流叠加，PDU-A 过载跳闸。

## 这个问题的本质

不是你不专业。是**人脑不适合同时追踪多个维度的约束**：

- U 位冲突？检查了。
- 线缆长度？检查了。
- 功率预算？……忘了。
- 浪涌电流？……没想到。
- 插座老化？……不知道。

每个维度单独看都简单，但组合起来，人的工作记忆不够用了。

## 现有的解决方式

### 方式一：人工校审

资深工程师逐条检查。有效，但：

- 成本高：校审人时费 = 设计人时费的 30-50%
- 不一致：不同校审人的严格程度不同
- 不可积累：校审经验随人退休而流失

### 方式二：Excel 检查表

把检查项做成表格，逐项打勾。比纯人工好，但：

- 静态：检查表更新后，历史项目不会自动复用新规则
- 孤立：Excel 里的数据和设计数据是两份，容易不一致
- 浅层：只能检查"填了没"，不能检查"对不对"

### 方式三：专业软件

BIM 平台、网管系统、计算软件。功能强大，但：

- 封闭：数据锁在专有格式里，无法版本控制
- 昂贵：license 费用高，小公司用不起
- 僵化：规则由软件商定义，用户无法自定义

## piki 的方式

**把检查规则写成代码，用 Git 管理，每次提交前自动运行。**

```python
# rules/power.py
from piki import rule, Context

@rule("TELECOM-POWER-001", "PDU 功率预算检查")
def check_pdu_budget(ctx: Context):
    """PDU 功率预算检查：负载不超过 80%"""
    for pdu in ctx.query("pdus"):
        devices = ctx.query("devices", pdu_id=pdu.id)
        load = sum(d.resolved.tdp_w for d in devices)
        ratio = load / pdu.resolved.capacity_w
        assert ratio <= 0.8, \
            f"{pdu.id} 负载率 {ratio:.1%}，超过 80% 阈值"
```

这条规则做了什么：

1. 遍历所有机柜和 PDU
2. 自动累加每个 PDU 上的设备功耗（从型号库读取 TDP）
3. 计算负载率，超过 80% 就报错

**你不需要记住检查功率预算。规则会记住。**

## piki 的核心价值

| 价值       | 说明                                                |
| ---------- | --------------------------------------------------- |
| **自动化** | 提交前自动运行所有规则，不依赖人的记忆              |
| **可积累** | 每次项目经验都可以写成规则，团队共享                |
| **可追溯** | 规则的变更历史在 Git 中，谁改了、为什么改，一目了然 |
| **可定制** | 国标、企标、项目特殊要求，都可以编码为规则          |
| **低成本** | 规则就是 Python 函数，不需要买 license              |

## piki 不做什么

piki 不是：

- **图形 CAD 软件**：不以画图/建模为主要交互方式，设计源头是文本
- **计算软件**：不做有限元分析，不做流体仿真
- **项目管理工具**：不管进度、成本、人力
- **权限系统**：不管谁有权限改什么（交给 Git）

piki 是 **Text-Native CAD**：一种**声明式建模**工具。你用文本声明设计意图，piki 负责解析、校验、生成预览。就像 Kubernetes 声明式管理容器，piki 声明式管理工程对象。

piki 只做一件事：**在设计数据提交前，自动检查规则，把问题找出来。**

## 适合谁用

- **有代码能力的工程师团队**：能写 Python，愿意把设计规则编码
- **规范化程度高的领域**：电信、数据中心、标准化建筑构件
- **需要多人协作的项目**：Git 工作流天然支持分支、评审、合并

## 不适合谁用

- **一次性、非标准化项目**：写规则的成本高于人工检查
- **没有代码能力的团队**：需要学习 Python 和 Git
- **高度创造性的设计阶段**：规则会束缚探索（但探索完成后、进入标准化阶段时，piki 很有用）
