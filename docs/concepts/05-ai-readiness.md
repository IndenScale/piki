# AI Readiness：为什么 Text-Native + 开源 是前提

> 不是"为了 AI 而 AI"，而是让 AI 能真正参与工程设计。

## 一个假设

假设你想让 AI 帮你检查一个机房扩容方案：

- **新增 2 台服务器**，放在 RACK-A01 的 U6-U7
- **接 PDU-A**，但你忘了算功率预算

这个方案可能记录在 Excel 里、画在图纸上、或者锁在某个闭源平台里。AI 能做什么？

| 数据形态           | AI 能做的事                        | 局限                                 |
| ------------------ | ---------------------------------- | ------------------------------------ |
| **手绘图纸**       | 几乎无法处理                       | 无结构化数据，OCR 也无法理解工程语义 |
| **Excel 表格**     | 读取单元格 → 猜测行列含义          | 格式自由、无 Schema、公式隐藏逻辑    |
| **AutoCAD DWG**    | OCR 识别图形 → 猜测设备位置        | 错误率高，无法校验                   |
| **闭源 SaaS 平台** | 调用 API（如果有）→ 受限于平台能力 | 黑盒，无法自定义规则                 |
| **piki YAML**      | 直接读取、理解、校验、生成         | **完整、精确、可复现**               |

Text-Native 不是目的，而是让 AI 能**可靠参与**的前提。下面展开说明：为什么手绘图纸和 Excel 不行。

---

## 为什么手绘图纸不行

手绘图纸是工程设计的原始形态，但它有几个致命问题：

### 1. 无结构化数据

图纸上的设备是一个**图形符号**，不是数据：

```
┌─────────────┐
│   [图标]    │  ← AI 看到的是一个矩形，不知道这是服务器还是交换机
│  SRV-01     │
└─────────────┘
```

AI 需要：

- OCR 识别文字（SRV-01）
- 视觉模型识别图标类型（服务器？交换机？）
- 猜测连接关系（这条线连到哪？）
- 猜测物理属性（功耗？高度？）

每一步都引入误差，累积后不可信。

### 2. 无法 diff

图纸改了，AI 怎么知道变了什么？

- 人眼对比两张 PDF？效率极低。
- 像素级 diff？格式转换、扫描质量、标注位置都会影响结果。
- 没有语义 diff：AI 不知道"PDU-A 负载从 550W 变成 950W"这个关键变化。

### 3. 无法自动化检查

图纸上的功率预算怎么检查？

- 没有结构化字段 → 无法累加
- 没有阈值定义 → 无法比较
- 没有引用关系 → 无法追踪

图纸适合**人看**，不适合**机器处理**。

### 4. 无法协作

两个人同时改一张图纸？

- 锁定文件，排队修改
- 或者各自画一版，最后人工合并
- 没有分支、没有评审、没有回滚

## 为什么 Excel 不行

Excel 比图纸进了一步——它有单元格、有公式。但远远不够：

### 1. 格式自由 = 语义模糊

同一类数据，每个人的表格结构不同：

```
Sheet A:        Sheet B:
┌──────┬─────┐  ┌──────┬──────────┐
│设备  │功耗 │  │服务器│功率(W)   │
├──────┼─────┤  ├──────┼──────────┤
│SRV-01│300  │  │01    │300       │
└──────┴─────┘  └──────┴──────────┘
```

AI 需要**猜测**：

- "设备"和"服务器"是同一列吗？
- "功耗"和"功率(W)"是同一含义吗？
- "01"是 "SRV-01" 的简写吗？

猜测 = 错误。

### 2. 公式隐藏逻辑

```excel
=SUM(B2:B10)*1.2  ← 为什么乘 1.2？是余量系数吗？
```

公式是黑盒：

- AI 读不到公式语义（"这是 TDP 余量"）
- 人也可能忘了当初为什么写 1.2
- 改了一个单元格，引用的公式可能不更新

### 3. 无版本控制

Excel 文件是二进制 blob：

- `git diff` 显示 `Binary files differ`，没有任何有用信息
- 无法分支、无法合并、无法代码评审
- "最终版*v3*修订*20240611*真的最终.xlsx"

### 4. 无引用完整性

Excel 里的 `rack_id` 怎么保证引用的机柜存在？

- 没有外键约束
- 没有自动校验
- 拼写错误（`RACK-A01` vs `RACK-AO1`）发现不了

Excel 适合**小规模、单人、一次性**的数据整理。工程设计需要**多人协作、持续迭代、自动校验**——Excel 做不到。

## AI Readiness 的四层基础

```text
┌─────────────────────────────────────────────┐
│  Layer 4: AI 应用层                          │
│  - 智能建议、自动修正、自然语言交互            │
├─────────────────────────────────────────────┤
│  Layer 3: 自动化层                           │
│  - CI/CD、pre-commit hook、定时巡检           │
├─────────────────────────────────────────────┤
│  Layer 2: 版本控制层                         │
│  - Git diff、分支评审、变更追溯               │
├─────────────────────────────────────────────┤
│  Layer 1: 数据层（Text-Native + 开源）        │
│  - YAML 设计文件、Python 规则、开放 Schema     │
└─────────────────────────────────────────────┘
```

没有 Layer 1-3，Layer 4 就是空中楼阁。

---

## Layer 1: Text-Native — AI 能读懂的数据

### LLM 原生理解文本

```yaml
# devices/SRV-03.yaml — LLM 可以直接读取、理解、生成
id: SRV-03
model: generic-server
rack_id: RACK-A01
position_u: 6
pdu_id: PDU-A
tdp_w: 400
```

LLM 看到这个文件，立刻知道：

- 这是一台服务器
- 放在 RACK-A01 的 U6 位置
- 接入 PDU-A
- 功耗 400W

不需要 OCR、不需要视觉模型、不需要解析专有格式。

### 结构化文本 > 自然语言

| 方式                                          | 问题                     |
| --------------------------------------------- | ------------------------ |
| "把服务器放在机柜 A01 的第六个 U 位"          | 歧义：U6 还是 U6-U7？    |
| `position_u: 6` + `height_u: 2`（来自 Model） | 精确：占用 U6-U7，无歧义 |

Text-Native 不是"用文本代替图形"，而是**用结构化声明代替模糊描述**。

---

## Layer 2: Git — AI 能理解的变更

### diff 是 AI 的母语

```diff
# git diff — LLM 可以直接分析变更意图
  id: SRV-03
  model: generic-server
  rack_id: RACK-A01
  position_u: 6
- pdu_id: PDU-A
+ pdu_id: PDU-B        # ← 改接 PDU，为什么？
  tdp_w: 400
```

AI 看到这段 diff，结合 commit message 和规则检查结果，可以：

- 理解变更原因（PDU-A 过载，改接 PDU-B）
- 检查是否引入新问题（PDU-B 容量是否足够？）
- 生成审核意见

### 没有 Git 的 AI = 瞎子

如果设计数据在闭源平台里：

- AI 看不到历史版本
- AI 不知道谁改了什么、为什么改
- AI 无法判断变更的合理性

Git 给 AI 提供了**时间维度**的上下文。

---

## Layer 3: CI/CD — AI 能验证的自动化

### 规则即契约

```yaml
# .github/workflows/piki.yml
- name: Run piki check
  run: piki check

- name: AI Review
  run: piki ai-review --model gpt-4
```

AI 生成的设计变更，必须通过：

1. **Schema 校验**（结构对不对）
2. **规则检查**（设计合不合理）
3. **AI 自审**（逻辑是否自洽）

这不是"AI 替代人"，而是"AI 先过机器关，人再终审"。

### 质量左移

```text
传统流程：设计 → 画图 → 评审 → 施工 → 发现问题 → 返工
piki 流程：设计 → 声明 → 自动检查 → 评审 → 施工
                ↑
                └── AI 在这里介入，成本最低
```

AI 在设计阶段发现问题，修复成本 = 改几行 YAML。
AI 在施工后发现问题，修复成本 = 拆机、换线、延期。

---

## Layer 4: AI 应用 — 不依赖黑盒

### 开源 = 可审计

piki 的规则是开源 Python：

```python
# rules/power.py — 你可以读、可以改、可以审计
@rule("TELECOM-POWER-001", "PDU 功率预算检查")
def check_pdu_budget(ctx: Context):
    threshold = ctx.config.get("power_threshold", 0.8)
    # ... 检查逻辑完全透明
```

AI 建议的方案，你可以追溯到每一条规则、每一个阈值、每一个数据来源。

### 拒绝黑盒锁定

| 模式                 | 问题                                                                   |
| -------------------- | ---------------------------------------------------------------------- |
| 闭源 SaaS + AI       | AI 怎么想的？不知道。数据在哪？厂商的服务器上。                        |
| **piki + 自托管 AI** | 规则透明、数据本地、AI 可替换（今天 GPT-4，明天 Claude，后天本地模型） |

Text-Native + 开源 = **AI 可插拔**。你不绑定任何特定模型、任何特定厂商。

---

## 对比总结

| 能力 | 手绘图纸 | Excel | 闭源 SaaS | piki (Text-Native) |
|------|---------|-------|----------|-------------------|
| **AI 可读性** | ❌ 无结构化数据 | ⚠️ 需猜测语义 | ⚠️ 依赖 API | ✅ 结构化 YAML，LLM 原生理解 |
| **版本控制** | ❌ 无法 diff | ❌ 二进制 blob | ⚠️ 厂商锁定 | ✅ Git 精确到字段 |
| **协作** | ❌ 文件锁定/排队 | ❌ 无法合并 | ⚠️ 账号权限 | ✅ Git 分支 + PR 评审 |
| **自动化检查** | ❌ 不可能 | ❌ 无 Schema/外键 | ⚠️ 平台定义规则 | ✅ Python 规则，无限定制 |
| **可审计** | ❌ 无逻辑可追溯 | ⚠️ 公式黑盒 | ❌ 黑盒 | ✅ 开源代码，完全透明 |
| **厂商锁定** | N/A | N/A | ❌ 严重 | ✅ 数据本地，工具开源 |

## 一句话总结

> **Text-Native 让 AI 能读懂设计，开源让 AI 能被审计，Git 让 AI 能理解变更，CI/CD 让 AI 能验证结果。**
>
> 手绘图纸和 Excel 的问题不是"不够先进"，而是**从一开始就不是为机器协作设计的**。AI 的参与只是让这些缺陷暴露得更彻底。
>
> 这四层不是"为了 AI"而存在的——它们本身就是好的工程实践。AI 只是让这些实践的价值被放大了。
