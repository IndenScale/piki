# piki 的生态位：敌、友与中间力量

> 公开说明自己和谁竞争、和谁互补、需要转化谁，本身就是一种策略。它能让盟友更快识别你，让中立者更快决定是否靠近你，也让团队自己更清楚边界在哪里。

---

## 为什么公开讨论生态关系

piki 不是要做一个“更好的 CAD”或“更好的 BIM”。它要改变的是工程设计的**入口层**和**真相源层**。这必然会牵动一批现有玩家的利益。

与其含糊其辞，不如把关系说清楚。公开生态定位有三个作用：

1. **对内统一决策**：哪些功能必须自建，哪些应该集成，哪些应该让路；
2. **对外降低交易成本**：潜在合作伙伴一眼看懂 piki 不碰什么、需要什么；
3. **建立叙事优势**：让市场知道，piki 不是在旧地图里抢地盘，而是在画新地图。

以下分析不是道德判断，而是基于**利益结构**和**能力边界**的判断。

## 逐类分析

### 1. CAD / BIM 平台：Autodesk、Dassault、Bentley

#### 关系：结构性替代对象 / 可共存的下游渲染器

他们现在的位置是“工程设计的入口和真相源”。piki 要争取的位置也是**真相源**。这是零和竞争。

但他们的优势是：

- 几十年的文件格式沉淀（DWG、RVT、DGN 等）；
- 嵌入企业工作流和审图流程；
- 强大的可视化、出图、协作功能。

piki 不打算在“渲染”和“出图”上打败他们。piki 的策略是：

> **让设计意图和规则校验在 piki 里完成，CAD/BIM 变成下游的可视化和交付工具。**

换句话说，Autodesk 可以从“设计操作系统”退化为“设计浏览器/渲染器”。如果它愿意开放接口、接受 piki 作为上游，它可以成为伙伴；如果它固守封闭生态，它就是被逐步绕过的对象。

**piki 的应对**：支持从这些格式导入，但绝不把 piki 的数据格式锁进他们的私有 schema。

---

### 2. 物理仿真厂商：Ansys、Synopsys、Cadence、Siemens EDA、Altair

#### 关系：互补型盟友 / 潜在利益错位者

piki 在前端做规则检查、约束校验、空间冲突检测；他们在后端做有限元、多物理场、电磁、热仿真。分工天然互补。

但利益上有摩擦：

- Ansys 按仿真核时和 License 收费，piki 把低级错误拦下来，可能减少无效仿真次数；
- 他们没有设计意图语义，很难自己下沉做 piki 的事；
- 如果他们收购或自建规则层，可能和 piki 形成竞争。

**最佳关系**：piki 作为 Ansys 的**前置质量门**，让每一次仿真都更有价值。Ansys 从“卖试错”转向“卖确定性”。

**piki 的应对**：保持开放接口，让仿真厂商容易接入；不试图替代他们的求解器；同时用开源规则库建立自己的不可替代性。

---

### 3. 型号 / 器件供应商：设备厂商、元器件平台、标准件库

#### 关系：天然盟友 / 数据供应者

piki 的规则能否准确，取决于模型数据是否准确：功耗、尺寸、接口类型、重量、热设计功耗、兼容性矩阵。

这些供应商最有动力提供准确、开放、可机器读取的型号数据。因为他们的产品被正确设计进去，意味着：

- 更少的选型错误；
- 更少的现场退货；
- 更强的生态绑定。

**piki 的应对**：

- 提供标准的型号数据 schema；
- 让供应商可以用 YAML/JSON 发布官方型号；
- 积极接受并合并型号供应商提交的官方型号数据与规则；
- 建立“官方认证型号库”，形成数据护城河。

---

### 4. 设计院 / 工程师：最终用户和转化对象

#### 关系：中间力量 / 需要争取的转化对象

他们不是敌人，也不是天然盟友。他们是：

- **当前既得利益者**：熟练使用 CAD/BIM，按图纸和工时计费；
- **未来被替代风险最大的一群**：大量重复性检查、绘图、出图工作会被 Agent 接管；
- **最具价值的设计判断力拥有者**：真正的工程经验、现场知识、复杂权衡能力。

piki 对他们的价值 proposition：

> **让工程师从“画图+校对的人”变成“定义规则+审核 Agent 输出”的人。**

但不是所有人都会接受。接受者通常是：年轻工程师、有代码能力的团队、对现有工具不满的人。

**piki 的应对**：

- 降低迁移成本：支持导入现有 CAD/BIM/Excel 数据；
- 尊重工作习惯：保留 GUI 预览和可视化；
- 提供清晰收益：减少返工、减少校审时间、提高设计权威性。

---

### 5. 云厂商 / AI Labs：AWS、Azure、GCP、OpenAI、Anthropic、国内大模型厂商

#### 关系：基础设施盟友

他们需要大量可被 Agent 操作的结构化数据和应用场景。piki 提供了一个**工程领域的 Agent 可操作 substrate**。

piki 对他们的价值：

- 一个垂直领域的高质量结构化数据集；
- 一个 LLM 可以读、写、验证的应用场景；
- 一个连接实体工程和 AI 的桥梁；
- **一个 RLVR（Reinforcement Learning from Verifiable Rewards，基于可验证奖励的强化学习）飞轮场景**：工程设计有明确的规则、约束和仿真结果，能为模型提供可验证的奖励信号，是继棋类、数学、代码之后，下一个可能规模化训练 Agent 的“硬约束”领域。

他们对 piki 的价值：

- 算力；
- 模型能力；
- **不只是分发渠道，更是模型能力进化的驱动力**：通过在工程场景中解决真实约束问题，模型学会长程规划、多目标权衡和物理一致性；
- 分发渠道。

**piki 的应对**：保持模型无关、厂商无关，让任何 AI 都能接入 piki 的 Agent API；同时用结构化规则为模型提供可验证的奖励环境，加速工程 Agent 的进化。

---

### 6. PLM / ERP / MES：Teamcenter、ENOVIA、SAP、Oracle

#### 关系：可合作的流程层伙伴 / 潜在边界冲突者

他们管理的是**产品全生命周期数据**、**物料**、**流程**、**制造执行**。piki 管理的是**设计意图和规则校验**。

理论上可以互补：piki 负责设计阶段，PLM 负责后续阶段。但边界上可能有摩擦：

- PLM 厂商可能也想把设计意图纳入自己的数据模型；
- piki 可能被视为侵入了 PLM 的地盘。

**piki 的应对**：明确不碰物料管理、生产排程、供应链管理；通过标准接口和 PLM 双向同步。

---

### 7. 开源社区 / 标准组织 / 学术界

#### 关系：规则生态盟友

国家标准、行业规范、学术研究成果，最终都可以编码为 piki 规则。开源社区可以帮助积累和审校这些规则。

**piki 的应对**：

- 开源核心和常见规则；
- 积极参与标准制定；
- 让学术界可以用 piki 做可复现的设计研究。

---

## piki 的策略原则

基于以上分析，piki 坚持以下原则：

1. **不碰渲染**：让 Autodesk/Blender/Omniverse 去卷可视化。
2. **不碰求解器**：让 Ansys/Synopsys 去卷物理仿真。
3. **不碰流程管理**：让 PLM/ERP 去卷物料和制造流程。
4. **占领设计意图层**：成为工程设计的**入口、真相源、规则层**。
5. **开放生态**：用开源规则库和标准接口吸引盟友，用不可替代的语义层形成护城河。

---

## 公开姿态

piki 对生态的公开姿态是：

> **我们不是来复制 AutoCAD 的，也不是来抢 Ansys 的饭碗。我们是来建立一种新的人机协同设计方式：人类定义意图和规则，Agent 在文本真相源上执行，传统工具作为可视化和仿真的下游。**

我们欢迎：

- 供应商提供官方型号数据；
- 仿真厂商接入 piki 作为前置检查；
- 云厂商和 AI Labs 基于 piki 构建工程 Agent；
- 设计院和工程师参与规则建设和审校。

我们警惕：

- 任何试图把设计真相源重新锁进私有格式的行为；
- 任何让 Agent 无法审计、无法回滚的“黑盒设计”模式。
