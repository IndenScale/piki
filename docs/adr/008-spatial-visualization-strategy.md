# ADR-008: 空间可视化策略——从文本到 3D

> 状态：已实现（基础完成）
> 日期：2026-06-12
> 作者：piki 核心团队

## 背景

piki 的核心理念是 Text-Native：工程师用 YAML 声明设计意图，规则引擎自动校验。但工程设计中的许多错误本质上是**空间问题**——设备 U 位重叠、管线干涉、维护通道不足——这些问题在文本中难以直观发现。

本 ADR 记录 piki 的空间可视化决策：需要什么、用什么技术、不做什么。

---

## 1. 为什么文本不够——空间直觉不可替代

| 场景 | 文本能否发现？ | 空间可视化能否发现？ |
|------|:---:|:---:|
| PDU 功率过载 | ✅ 数值计算 | 不必要 |
| 两台设备 U 位重叠 | ⚠️ 规则检查可以 | ✅ 更直观 |
| 设备与机柜后门干涉 | ❌ 难以表达 | ✅ 一眼可见 |
| 液冷管路弯折半径不足 | ❌ 文本难以描述 | ✅ 必须可视化 |

工程设计的错误成本极高。空间直觉在设计评审、施工交底、运维定位中仍然非常重要。piki 需要一种机制，让文本设计自动派生出可感知的三维视图。

### 1.1 核心定位：沟通媒介而非设计工具

piki 不打算成为交互式 CAD 软件。3D 视图是**只读的沟通媒介**：设计阶段写 YAML → 评审阶段看 3D → 发现问题改 YAML → 重新生成。修改设计的唯一途径是改 YAML。

---

## 2. 为什么选择 OpenUSD 而非直接用 Three.js

Three.js 是优秀的 Web 3D 渲染库，但直接用 Three.js 作为 piki 的"原生"场景格式会带来问题：

| 维度 | Three.js 原生方案 | OpenUSD |
|------|------------------|---------|
| 场景描述能力 | JSON 场景图，无组合/引用/变体 | 层叠编辑、外部引用、变体集、时间采样 |
| 工业生态兼容性 | Three.js 是渲染库，非交换格式 | NVIDIA Omniverse、Apple、Autodesk、Blender 原生支持 |
| 物理/碰撞标准 | 无，需自行设计 | UsdPhysics 是跨工具标准 |
| 长期归档 | Three.js 版本迭代快，格式不保证兼容 | USD 是 ISO 标准化方向，向后兼容 |
| 多工具协作 | 导出到 BIM/数字孪生需自定义胶水 | BIM 平台、数字孪生可直接读取 USD |

**关键洞察**：Three.js 是渲染器，OpenUSD 是场景描述系统。piki 的核心价值在于设计意图的声明与校验，需要一种能够承载设计意图、被工业工具链消费的场景格式。OpenUSD 是唯一满足这一要求的选择。

### 2.1 Web 渲染：glTF 过渡

OpenUSD 的 WASM API（v26.03 刚发布）尚未稳定。在浏览器中，piki 通过 USD → glTF 转换实现渲染。Piki Studio 使用自研 USDA 文本解析器渲染代理几何，无需 WASM。未来 WASM 稳定后可迁移到浏览器原生 USD 渲染。

## 3. 碰撞检测：纯几何算法，不依赖物理引擎

碰撞检测在 piki 规则引擎中以纯 Python AABB/OBB 算法实现：

```python
@rule("TELECOM-COLLISION-001", "3D 空间碰撞检查")
def check_3d_collision(ctx):
    for i, (id1, box1) in enumerate(boxes):
        for id2, box2 in boxes[i+1:]:
            if intersect(box1, box2):
                assert False, f"设备 {id1} 与 {id2} 空间冲突"
```

碰撞结果作为元数据嵌入 USD，供预览器高亮显示。

**理由**：几何碰撞算法轻量、可预测、无外部依赖；不绑定到特定物理引擎（PhysX、Bullet 等）；结果嵌入 USD 后任何 viewer 都能看到高亮。

### 3.1 不做什么

| 排除项 | 理由 |
|--------|------|
| 实时交互式编辑器 | 违背 Text-Native 哲学 |
| 浏览器原生 USD 渲染（当前） | 等待 OpenUSD WASM 稳定，glTF 过渡 |
| 完整物理仿真 | 刚体动力学、流体超出工程设计校验范围 |
| PBR 材质管线 | 工程预览以空间关系为主，基础材质足够 |

## 4. 技术选型总结

| 组件 | 选型 | 理由 |
|------|------|------|
| 场景描述格式 | OpenUSD (USDA/USDC) | 工业标准、生态丰富、长期归档 |
| Python USD 库 | `usd-core` (PyPI) | 官方维护，无需编译 C++ |
| Web 渲染格式 | glTF 2.0（过渡） | Three.js 原生支持，生态成熟 |
| Web 渲染引擎 | Three.js | 最成熟的 Web 3D 库 |
| 几何碰撞检测 | 纯 Python AABB/OBB | 轻量、无依赖、结果可嵌入 USD |
| 代理几何生成 | 程序生成 Box/Cylinder | 无 USD 资产时自动降级 |

---

## 5. 决策总结

| 决策 | 选择 | 核心理由 |
|------|------|----------|
| 3D 定位 | 只读沟通媒介，非设计工具 | 修改设计只改 YAML，不为图形破坏文本真相源 |
| 场景格式 | OpenUSD | 工业标准、组合/引用/变体、跨工具协作 |
| Web 渲染 | glTF 过渡，自研 USDA 解析器 | 无需 WASM，启动快，当前阶段足够 |
| 碰撞检测 | 纯 Python 几何算法 | 轻量、无依赖、结果嵌入 USD |
| 不做什么 | 交互编辑、完整物理仿真、PBR | 边界清晰，核心不膨胀 |
