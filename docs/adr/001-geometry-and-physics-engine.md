# ADR-001: 几何引擎与物理引擎的选型与集成策略

> 状态：已接受  
> 日期：2026-06-11  
> 作者：piki 核心团队

## 背景

piki 是一个 Text-Native CAD 框架，核心理念是"设计的本质是决策，不是画图"。工程师用 YAML 声明设计意图，规则引擎自动校验数据完整性、规范符合性和设计合理性。

随着项目演进，一个反复出现的问题浮现：**仅有文本和表格，是否足以支撑工程设计的全生命周期？**

本 ADR 记录我们对几何引擎、物理引擎必要性的思考，以及为什么选择在工具链的特定位置集成 OpenUSD，而非直接在 Three.js 中重建预览能力。

---

## 1. 必要性：为什么需要几何/物理引擎

### 1.1 空间直觉在工程设计中不可替代

工程设计的错误成本极高：

- 结构梁配筋不足 → 返工、延期、安全隐患
- 电信设备 PDU 超容 → 宕机、业务中断
- **机柜内设备 U 位重叠** → 现场无法安装
- **方舱间管线干涉** → 现场切割、焊接、返工
- **设备与门框/通道碰撞** → 运输受阻、吊装失败

这些问题的共同特征是：**它们发生在三维空间中，文本描述无法直接触发人类的空间直觉。**

| 场景                     | 文本能否发现？      | 空间可视化能否发现？ |
| ------------------------ | ------------------- | -------------------- |
| PDU 功率过载             | ✅ 可以（数值计算） | ❌ 不必要            |
| 外键引用错误             | ✅ 可以（关系检查） | ❌ 不必要            |
| **两台设备 U 位重叠**    | ⚠️ 可以（规则检查） | ✅ **更直观**        |
| **设备与机柜后门干涉**   | ❌ 难以表达         | ✅ **一眼可见**      |
| **液冷管路弯折半径不足** | ❌ 文本难以描述     | ✅ **必须可视化**    |
| **方舱间净距不足**       | ❌ 文本难以描述     | ✅ **必须可视化**    |

**结论**：空间直觉在工程设计、沟通、交底中仍然非常重要。文本是"真相源"，但空间可视化是"理解加速器"。piki 需要一种机制，让文本设计自动派生为可感知的三维视图。

### 1.2 "沟通媒介"比"设计工具"更重要

piki 不打算成为交互式 CAD 软件（如 SolidWorks、Revit）。我们的定位是：

- **设计阶段**：工程师在 YAML 中声明意图，规则引擎校验
- **评审阶段**：同事通过 3D 预览快速理解空间关系
- **交底阶段**：施工方通过 Walkthrough 确认通道、吊装路径、维护空间
- **运维阶段**：值班工程师通过可视化界面定位故障设备

在这些场景中，3D 视图是**只读的沟通媒介**，不是**可编辑的设计工具**。修改设计？改 YAML，重新生成。

### 1.3 碰撞检查需要几何计算

piki 的规则引擎已经支持跨记录业务规则（如 PDU 功率预算、U 位冲突）。但 U 位冲突只是 1D 碰撞检查，真实的工程问题需要 3D 碰撞检查：

- 设备深度是否超出机柜？
- 液冷快接头与相邻设备间距是否足够？
- 方舱门开启时是否与其他方舱干涉？
- 维护通道宽度是否满足规范？

这些问题需要**几何引擎**计算包围盒、距离、相交；更复杂的问题需要**物理引擎**模拟刚体运动（如门扇开启轨迹）。

## 2. 可行性：为什么使用 OpenUSD

### 2.1 为什么不直接用 Three.js 做预览器

Three.js 是优秀的 Web 3D 渲染库，但直接用 Three.js 作为 piki 的"原生"场景格式会带来以下问题：

| 维度               | Three.js 原生方案                       | OpenUSD 方案                                                |
| ------------------ | --------------------------------------- | ---------------------------------- |
| **场景描述能力**   | 弱：JSON 格式的场景图，无组合/引用/变体 | **强**：层叠编辑、外部引用、变体集、时间采样                |
| **工业生态兼容性** | 低：Three.js 是渲染库，非交换格式       | **高**：NVIDIA Omniverse、Apple、Autodesk、Blender 原生支持 |
| **物理/碰撞标准**  | 无：需自行设计数据结构                  | **有**：UsdPhysics 是跨工具标准                             |
| **长期归档**       | 差：Three.js 版本迭代快，格式不保证兼容 | **好**：USD 是 ISO 标准化方向，向后兼容承诺                 |
| **多工具协作**     | 差：导出到 BIM/数字孪生需自定义胶水     | **好**：BIM 平台、数字孪生可直接读取 USD                    |
| **文件规模**       | 差：大场景 JSON 臃肿                    | **好**：USDC 二进制高效，支持延迟加载                       |

**关键洞察**：Three.js 是**渲染器**，OpenUSD 是**场景描述系统**。piki 的核心价值在于"设计意图的声明与校验"，我们需要一种能够**承载设计意图**
并**与外部工具互操作**的场景格式，而不仅仅是能在浏览器里画出来。

类比：

- Three.js ≈ PDF 阅读器：能看，但不能编辑，也不能被其他工具理解内容
- OpenUSD ≈ Markdown + Git：文本是真相源，可以渲染为 HTML，也可以被任何工具读取

### 2.2 Model 如何集成 USD 资产

piki 的三层数据模型（Family → Model → Instance）为 USD 资产的集成提供了自然的位置：

```
Family（约束结构）
  └── Model（默认值 + USD 资产引用）
        └── Instance（实际部署值，可覆盖）
```

**在 Model 层引用 USD 资产**：

```yaml
# library/devices/dell-r740.yaml
model: dell-r740
family: ServerFamily

physical:
  height_u: 2
  depth_mm: 715
  width_mm: 445

power:
  tdp_w: 350
  psu_count: 2

# USD 资产引用 —— 可选，无则使用程序生成的代理几何
assets:
  usd:
    # 方式 A：引用外部 USD 文件（推荐）
    reference: "./assets/dell-r740.usda"
    # 方式 B：内联简单几何（快速原型）
    inline:
      type: "box"
      size: [0.445, 0.0889, 0.715] # 宽 × 高 × 深（米）
    # 方式 C：指向厂商提供的 USDZ
    usdz: "https://vendor.example.com/dell-r740.usdz"
```

**设计原则**：

1. **渐进式**：没有 USD 资产的 Model，用程序生成的 Box/Cylinder 代理几何
2. **可覆盖**：Instance 可覆盖 Model 的资产引用（如现场使用定制支架）
3. **可离线**：USD 引用支持本地路径和 HTTP URL，构建时缓存到项目目录
4. **可降级**：USD 资产加载失败时，自动回退到代理几何，不中断流程

**关键设计决策**：

1. **USD 导出是 Generator，不是核心引擎的一部分**
   - 理由：保持核心轻量，USD 依赖设为 optional
   - 用户不装 `usd-core` 也能用 `piki check`

2. **预览是独立命令，不是 check 的副作用**
   - `piki check` 只做校验，不启动任何图形界面
   - `piki preview` 或 `piki generate usd` 才产生可视化产物
   - 理由：CI/CD 环境中不应依赖图形库

3. **Web 预览通过静态站点生成，不是实时服务器**
   - `piki generate site` 输出纯 HTML/JS/GLB 文件
   - 部署到 GitHub Pages / Cloudflare Pages / S3
   - 理由：零运维成本，与 piki 的 Git-Native 工作流一致

## 3. 范围决策：我们只做 1、3、4

### 3.1 我们做什么

| 编号  | 功能                 | 说明                                      | 优先级 |
| ----- | -------------------- | ----------------------------------------- | ------ |
| **1** | **USD 场景导出**     | `piki generate usd-scene` 生成 USDA/USDC  | P0     |
| **3** | **静态网页部署**     | `piki generate site` 输出可部署的静态站点 | P1     |
| **4** | **Walkthrough 支持** | 预定义相机路径，自动漫游场景              | P1     |

### 3.2 我们不做（明确排除）

| 编号  | 功能                      | 不做理由                                                                     |
| ----- | ------------------------- | ---------------------------------------------------------------------------- |
| **2** | **实时交互式编辑器**      | 违背 Text-Native 哲学。修改设计？改 YAML，重新生成。                         |
| **5** | **浏览器内原生 USD 渲染** | 等待 OpenUSD 官方 WASM API 稳定（v26.03 刚发布）。短期用 glTF 过渡。         |
| **6** | **完整物理仿真**          | 刚体动力学、流体、柔体超出工程设计校验范围。碰撞检测用几何算法即可。         |
| **7** | **PBR 材质管线**          | 工程预览以空间关系为主，不需要电影级材质。USD Preview Surface 基础材质足够。 |

### 3.3 碰撞检查的特殊处理

碰撞检查**不依赖 USD 物理引擎**，而是在 piki 规则引擎中以纯几何算法实现：

```python

def check_3d_collision(ctx):

            if intersect(box1, box2):

```

USD 导出时**附加碰撞结果作为元数据**，供预览器高亮显示：

```usda
# 碰撞结果以 customData 或 primvars 附加
def Cube "SRV-01" (
    prepend apiSchemas = ["PhysicsCollisionAPI"]
)
{
    # piki 生成的碰撞标记
    bool piki:collision = true
    string piki:collisionWith = "SRV-02"
    color3f[] primvars:displayColor = [(1, 0, 0)]  # 红色高亮
}
```

**理由**：

- 几何碰撞算法轻量、可预测、无外部依赖
- 结果嵌入 USD 后，任何 USD viewer 都能看到高亮
- 不绑定到特定物理引擎（PhysX、Bullet 等）

## 4. 技术选型总结

| 组件              | 选型                            | 理由                          |
| ----------------- | ------------------------------- | ----------------------------- |
| **场景描述格式**  | OpenUSD (USDA/USDC)             | 工业标准、生态丰富、长期归档  |
| **Python USD 库** | `usd-core` (PyPI)               | 官方维护，无需编译 C++        |
| **Web 渲染格式**  | glTF 2.0 (过渡)                 | Three.js 原生支持，生态成熟   |
| **Web 渲染引擎**  | Three.js                        | 最成熟的 Web 3D 库，glTF 原生 |
| **静态站点部署**  | GitHub Pages / Cloudflare Pages | 零运维，与 Git 工作流天然结合 |
| **几何碰撞检测**  | 纯 Python AABB/OBB              | 轻量、无依赖、结果可嵌入 USD  |
| **代理几何生成**  | 程序生成 Box/Cylinder           | 无 USD 资产时自动降级         |
