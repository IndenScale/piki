# ADR-006: Piki Studio — Web 版工程设计 IDE

> 状态：已接受  
> 日期：2026-06-11  
> 作者：piki 核心团队

---

## 背景

piki 的核心理念是"Text-Native CAD"——用文本声明设计意图，用规则引擎自动校验。但工程设计中的许多错误本质上是**空间问题**：

- 机柜内设备 U 位重叠 → 现场无法安装
- 管线与结构梁干涉 → 现场切割返工
- 设备与通道碰撞 → 运输受阻

这些问题在文本中难以直观发现。工程师需要一种方式，在提交设计前**快速预览三维布局**，并在发现问题后直接回到文本修正。

此前 piki 提供 `piki preview --web` 命令，基于 Autodesk 的 USD WASM fork 实现了一个轻量 USDA 查看器。但该方案存在根本性问题，促使我们重新设计预览体验。

---

## 1. 旧方案的局限性

### 1.1 功能割裂

旧预览器是一个**独立的 USDA 文件查看器**，与 piki 的项目结构无关：

| 能力 | 旧预览器 | 工程师实际需求 |
|------|---------|-------------|
| 打开方式 | 拖拽 USDA 文件 | 打开整个 piki 项目 |
| 左侧面板 | 无 | 显示 YAML 文件树和集合层级 |
| 选中反馈 | 仅 3D 高亮 | 同步高亮 + 显示属性 |
| 规则结果 | 看不到 | 在状态栏显示 check 结果 |
| 编辑闭环 | 无 | 改 YAML → 刷新预览 |

### 1.2 技术债务

旧预览器依赖 `needle-tools/usd-viewer` 的完整仓库（含 Fastify 服务器、WASM 绑定、示例文件），体积超过 30MB。其中：

- `usd-wasm/` 包含 16MB 的 WASM 二进制
- `public/` 包含大量示例 USDZ 文件和 HDR 环境贴图
- `server.js` 仅用于提供 COOP/COEP headers

这些资源与 piki 的核心功能无关，却随 Python 包一起分发。

### 1.3 维护困难

旧预览器是第三方项目的 fork，其 JavaScript 代码与 piki 的 Python 代码无直接交互。任何功能扩展（如显示实例属性、集成 check 结果）都需要在两个独立技术栈之间做桥接，复杂度极高。

---

## 2. 新方案：Piki Studio

### 2.1 定位

Piki Studio 是一个**基于浏览器的工程设计 IDE**，不是独立的查看器。它与 piki 项目深度集成：

- 直接读取本地 piki 项目目录（通过 File System Access API）
- 解析 YAML 实例文件，构建场景树
- 解析 USDA 场景文件，渲染 3D 预览
- 选中对象时同步显示解析后的属性

### 2.2 架构

```
┌─────────────────────────────────────────────────────────────┐
│                        Piki Studio                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │   Toolbar   │  │  SceneTree  │  │   PropertiesPanel   │  │
│  │  打开项目    │  │  资产层级   │  │    属性面板         │  │
│  └─────────────┘  └──────┬──────┘  └─────────────────────┘  │
│                          │                                   │
│  ┌───────────────────────┼─────────────────────────────┐    │
│  │                   Viewport                          │    │
│  │              Three.js 3D 渲染                       │    │
│  │         USDA 解析 → Mesh → 场景渲染                  │    │
│  └───────────────────────┼─────────────────────────────┘    │
│                          │                                   │
│  ┌───────────────────────┴─────────────────────────────┐    │
│  │                   StatusBar                          │    │
│  │         项目名 | 消息 | 检查统计                      │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| 语言 | TypeScript | 类型安全，IDE 开发体验好 |
| 构建工具 | Vite | 快速，原生 ESM，配置简单 |
| 3D 渲染 | Three.js | 成熟，社区大，USD 生态已有集成 |
| USDA 解析 | 文本解析器（自研） | 不依赖 WASM，启动快，足够渲染代理几何 |
| 项目读取 | File System Access API | 浏览器原生，无需后端服务器 |

**为什么不用 USD WASM？**

当前阶段，Piki Studio 只需要渲染代理几何（Box）和显示属性。自研的 USDA 文本解析器（约 200 行）已足够满足需求，且：

- 无需下载 16MB WASM 二进制
- 无需 COOP/COEP headers（简化部署）
- 解析速度快，无初始化延迟

未来如需渲染复杂 USD 场景（引用外部资产、材质、动画），可再集成 USD WASM。

### 2.4 与 piki CLI 的关系

| 场景 | CLI | Studio |
|------|-----|--------|
| 运行检查 | `piki check` | 未来可集成 |
| 生成 USD | `piki generate usd-scene` | 读取生成的 scene.usda |
| 查看 3D | ❌ 无 | ✅ 打开项目即看 |
| 编辑 YAML | VS Code / Vim | 未来可支持 |

Piki Studio 是 CLI 的**可视化补充**，不是替代。设计的源头仍然是文本，Studio 只是让"看效果"这件事更直观。

---

## 3. 删除旧预览器的理由

### 3.1 功能重叠且更弱

旧预览器能做的事（渲染 USDA），Studio 都能做；Studio 能做的事（项目级浏览、属性面板、场景树），旧预览器都做不了。

### 3.2 减少维护负担

删除 `src/piki/ext/usd/web-viewer/` 后：

- 移除 30MB+ 的静态资源
- 移除 `piki preview` CLI 命令及相关代码
- 移除 `web_server.py` 和 Fastify 依赖
- Python 包体积显著减小

### 3.3 统一技术栈

旧预览器是 JavaScript（外部 fork），Studio 是 TypeScript（自研）。统一为自研 TypeScript 后：

- 功能演进完全可控
- 与 piki 数据模型（`PikiInstance`、`SceneObject`）直接对应
- 无需维护 Python-JS 桥接代码

---

## 4. 决策

**接受**以下变更：

1. 新增 `studio/` 目录，包含 Piki Studio 的完整 TypeScript 源码
2. 删除 `src/piki/ext/usd/web-viewer/` 目录及其所有内容
3. 删除 `src/piki/commands/preview.py` 和 CLI 中的 `preview` 子命令
4. 删除 `src/piki/ext/usd/web_server.py`
5. 更新 `.gitignore`，排除 `studio/node_modules/` 和 `studio/dist/`
