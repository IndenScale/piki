# ADR-002: 文本格式与"一实例一文件"的数据组织方式

> 状态：已接受  
> 日期：2026-06-11  
> 作者：piki 核心团队

## 背景

piki 选择 YAML 作为设计数据的存储格式，并坚持"一个实例一个文件"（one instance per file）的组织方式。这个决策看似简单，但背后涉及版本控制、协作效率、AI 可读性、工具链集成等多个维度的权衡。本 ADR 记录这些决策的理由。

---

## 1. 为什么使用文本格式（而非二进制或数据库）

### 1.1 文本是工程协作的通用语言

工程设计数据需要经过多个环节：设计 → 评审 → 交底 → 施工 → 运维。每个环节使用不同的工具，由不同的角色参与。文本是唯一能被**所有工具**和**所有角色**共同理解的基础格式。

| 维度           | 二进制格式（DWG/Revit/RVT） | 数据库（SQL/NoSQL）    | 文本格式（YAML/JSON/TOML） |
| -------------- | --------------------------- | ---------------------- | -------------------------- |
| **人类可读**   | ❌ 需要专用软件打开         | ⚠️ 需查询工具          | ✅ 任何文本编辑器          |
| **版本控制**   | ❌ `git diff` 无意义        | ⚠️ 需 schema migration | ✅ 精确到字段的 diff       |
| **代码评审**   | ❌ 无法行级评审             | ⚠️ 数据变更难理解      | ✅ 像审代码一样审设计      |
| **CI/CD 集成** | ❌ 需专有 SDK               | ⚠️ 需连接串            | ✅ 直接读取、解析          |
| **AI 可读性**  | ❌ 需逆向工程               | ⚠️ 需 schema 描述      | ✅ LLM 原生理解            |
| **长期归档**   | ⚠️ 依赖软件版本             | ✅ 稳定                | ✅ 稳定                    |
| **工具锁定**   | ❌ 强锁定                   | ⚠️ 中等锁定            | ✅ 无锁定                  |

### 1.2 为什么是 YAML 而非 JSON 或 TOML

piki 的配置文件（`piki.toml`）使用 TOML，因为 TOML 适合**键值对配置**。但设计数据（实例/型号）使用 YAML，因为：

| 特性               | YAML        | JSON          | TOML                 |
| ------------------ | ----------- | ------------- | -------------------- | --------------- |
| **注释支持**       | ✅ `# 注释` | ❌ 不支持     | ✅ `# 注释`          |
| **多行字符串**     | ✅ `\|` 管道符      | ⚠️ `\n` 转义         | ⚠️ `"""` 三引号 |
| **列表表达**       | ✅ `- item` | ⚠️ `["item"]` | ⚠️ `[[item]]` 嵌套表 |
| **嵌套结构可读性** | ✅ 缩进直观 | ⚠️ 括号嵌套   | ⚠️ 表头嵌套          |
| **尾随逗号**       | ✅ 允许     | ❌ 不允许     | ✅ 允许              |
| **人类书写友好度** | ⭐⭐⭐      | ⭐⭐          | ⭐⭐⭐               |

YAML 的缩进语法与 Python 类似，对工程师群体最自然。多行字符串（如设备描述、备注）在 YAML 中书写最方便。

**代价**：YAML 的灵活性也带来了歧义（如 `yes` 被解析为布尔值 `true`）。piki 通过 pydantic Family 的严格类型校验来消解这一风险。

### 1.3 Text-Native 不是"反图形"

一个常见的误解是：Text-Native = 不用图形。这是错误的。

```
Text-Native 的正确理解：

文本是设计的"唯一真相源"（Single Source of Truth）
    ↓
预览、BOM、碰撞检查、3D 视图都是"派生视图"
    ↓
修改设计？改文本，重新生成所有派生视图
```

类比：

- Terraform 用 HCL 声明基础设施，AWS 控制台是只读视图
- Kubernetes 用 YAML 声明 Pod 配置，Dashboard 是只读视图
- piki 用 YAML 声明工程对象，USD 预览 / 静态站点是只读视图

图形界面用于**理解**和**沟通**，文本用于**定义**和**版本控制**。两者不是替代关系，而是分层关系。

## 2. 为什么坚持"一实例一文件"

### 2.1 问题：如果所有实例放在一个文件里会怎样？

```yaml
# devices.yaml —— 反模式：所有设备放在一个文件
- id: SRV-01
  rack_id: RACK-A01
  position_u: 10
- id: SRV-02
  rack_id: RACK-A01
  position_u: 8
- id: SRV-03
  rack_id: RACK-A01
  position_u: 6
```

**问题**：

1. **Git diff 噪音**：新增一台设备，diff 显示整个文件变更，无法快速定位"改了哪台设备"
2. **合并冲突**：多人同时编辑不同设备，Git 无法自动合并（YAML 列表冲突难以解析）
3. **权限粒度粗**：无法按设备分配编辑权限（虽然 Git 本身不控制权限，但文件粒度影响代码所有者配置）
4. **文件锁定**：某些协作模式下，整个文件被锁定，其他人无法编辑无关设备
5. **历史追溯困难**：`git blame` 显示的是文件级别的变更，无法直接看到"SRV-03 是谁创建的"

### 2.2 "一实例一文件"的优势

```
devices/
├── SRV-01.yaml    ← 独立文件，独立历史
├── SRV-02.yaml
└── SRV-03.yaml
```

| 优势             | 说明                                                     |
| ---------------- | -------------------------------------------------------- |
| **精确 diff**    | `git diff` 只显示变更的设备文件，一目了然                |
| **无合并冲突**   | 不同设备由不同文件表示，Git 自动合并                     |
| **独立历史**     | `git blame devices/SRV-03.yaml` 直接追溯该设备的完整历史 |
| **独立评审**     | PR 中只评审变更的设备，不受无关变更干扰                  |
| **独立权限**     | CODEOWNERS 可以按设备类型/目录分配审核人                 |
| **独立生命周期** | 设备退役 = 删除文件，不影响其他设备                      |
| **独立引用**     | 外部系统引用特定设备时，使用文件路径即可定位             |
| **IDE 资产树映射** | 文件目录树直观对应工程资产树，开发者可以在 IDE 的目录面板中直接浏览整个工程结构，文件即资产 |

### 2.3 与 folder-db 的渊源

piki 的数据组织方式深受 [folder-db](https://github.com/indenscale/folder-db) 影响：

```
folder-db 理念：
- 文件系统就是数据库
- 目录就是表
- 文件就是记录
- 文件名就是主键
```

piki 继承了这一理念，但增强了：

- **Family 约束**：folder-db 只有类型声明，piki 有 pydantic 强校验
- **Model 默认值**：folder-db 无型号库概念，piki 支持 Model → Instance 的继承覆盖
- **规则引擎**：folder-db 无跨记录检查，piki 有 pytest 风格规则

### 2.4 代价与缓解

| 代价                                    | 缓解措施                                                               |
| --------------------------------------- | ---------------------------------------------------------------------- |
| 文件数量多（1000 台设备 = 1000 个文件） | 文件系统天然支持大量小文件；按目录组织（`devices/rack-a01/`）          |
| 批量编辑困难                            | `sed` / `jq` / Python 脚本批量处理；piki 未来可能提供 `piki bulk-edit` |
| 文件系统性能                            | 现代文件系统（ext4/APFS）处理 10K+ 文件无压力；必要时可按目录分片      |
| 跨文件引用校验                          | piki 规则引擎自动检查外键完整性（`rack_id` 引用的机柜是否存在）        |

## 3. 为什么一个型号一个文件

型号库（`library/`）同样遵循"一型号一文件"：

```
library/
└── devices/
    ├── generic-server.yaml
    ├── dell-r740.yaml
    └── hp-dl380.yaml
```

理由与实例相同，但额外有：

1. **型号独立演进**：Dell 发布 R740 新固件，只需更新 `dell-r740.yaml`，不影响其他型号
2. **型号独立分发**：团队可以共享特定型号文件，无需导出整个型号库
3. **型号独立审核**：厂商规格变更需要审核，独立文件便于 CODEOWNERS 配置

---

## 4. 决策总结

| 决策         | 选择               | 核心理由                                      |
| ------------ | ------------------ | --------------------------------------------- |
| **数据格式** | YAML               | 人类书写友好、注释支持、多行字符串、嵌套可读  |
| **组织方式** | 一实例一文件       | Git 精确 diff、无合并冲突、独立历史、独立评审 |
| **型号组织** | 一型号一文件       | 独立演进、独立分发、独立审核                  |
| **配置格式** | TOML (`piki.toml`) | 键值对配置更简洁，与实例数据区分              |

## 5. 反模式警示

以下做法与 piki 的设计哲学相悖：

```yaml
# ❌ 反模式 1：所有实例放在一个文件
devices.yaml:
  - id: SRV-01
  - id: SRV-02

# ❌ 反模式 2：使用 JSON（无注释、尾随逗号严格）
devices/SRV-01.json:
  {"id": "SRV-01", "rack_id": "RACK-A01"}

# ❌ 反模式 3：文件名不含 ID（无法快速定位）
devices/
  ├── 001.yaml    # 不知道这是什么设备
  └── 002.yaml

# ✅ 正确做法
devices/
  ├── SRV-01.yaml
  └── SRV-02.yaml
```
