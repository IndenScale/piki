# piki 动议（Initiatives）

本目录记录 piki 的中长期发展目标。每个动议（Initiative）描述一个需要持续投入的战略方向，而不是一次性的功能需求。

动议与 ADR、RFC 的区别：

| 文档类型 | 回答的问题 | 生命周期 |
|---|---|---|
| **Initiative（动议）** | 我们要往哪个方向走？为什么值得做？ | 长期，直到目标达成或战略调整 |
| **RFC** | 具体方案怎么设计？接口长什么样？ | 方案确定后归档 |
| **ADR** | 某个关键决策为什么选择 A 而不是 B？ | 决策落地后归档 |

## 动议之间的关系

```text
┌─────────────────────────────────────────────────────────────┐
│                    SD-HWE-Bench (001)                       │
│              衡量工程 Agent 设计能力的竞技场                  │
└───────────────────────────┬─────────────────────────────────┘
                            │ 需要可复用的部件与可验证的设计声明
        ┌───────────────────┴───────────────────┐
        ▼                                       ▼
┌───────────────┐                     ┌───────────────────────┐
│  EPM (002)    │                     │ GitHub 即工程设计 (003)│
│ 部件注册与依赖 │◄───────────────────►│  ADL + Git + CI/CD    │
└───────┬───────┘                     └───────────┬───────────┘
        │                                         │
        │    ┌────────────────────────────────────┘
        │    │
        ▼    ▼
┌─────────────────────────────┐
│ 质量左移工具链 (004)        │
│ 语法 → 静态分析 → 设计规则校验 │
└─────────────────────────────┘
```

- **EPM (002)** 为 **GitHub 即工程设计 (003)** 提供大文件（几何、网格、图纸）的存储与版本化能力，避免 Git LFS。
- **GitHub 即工程设计 (003)** 为 **SD-HWE-Bench (001)** 提供数据集和任务的可复现基础：每个任务都是可被 `git clone`、可被 `piki check` 的声明式工程仓库。
- **质量左移工具链 (004)** 是 **piki check** 的核心能力，服务于 EPM 包质量门、GitHub CI/CD 质量检查、以及 SD-HWE-Bench 的自动评分。

## 如何新增动议

1. 在本文档表格中预留下一个编号；
2. 复制 [001-sd-hwe-bench.md](001-sd-hwe-bench.md) 的模板；
3. 填写标题、状态、日期、作者、摘要、背景、目标、范围、实现路径、风险与下一步；
4. 更新本 README 的链接与关系图。
