# HDA：Hardware Design Automation，实体工程的设计自动化赛道

> SDH 是一种思想，HDA 是一个赛道；piki 是这条赛道上的开源参考实现。
>
> 本文解释 HDA 与 EDA、CAD/BIM、SDH 的关系，以及为什么 piki 要让构建 HDA 应用变得像搭积木一样简单。

---

## 一、HDA 是什么

**HDA（Hardware Design Automation）= 用 EDA 的心智模型，为机械、建筑、通信、能源等非电子硬件领域，构建一套“声明式建模 + 自动规则检查 + 自动交付物生成”的设计自动化系统。**

EDA 解决了芯片和 PCB 的设计自动化问题：工程师用 Verilog 写设计意图，综合器、DRC、LVS、STA 自动验证，最后输出网表和版图。HDA 要把这套方法论推广到更广阔的物理工程领域：

- 电信机房里的机柜、PDU、交换机、服务器
- 数据中心里的方舱、UPS、冷通道、线缆
- 机械产品里的零件、装配、干涉、公差
- 建筑暖通里的风管、水管、设备、能耗

这些领域的共同点是：设计意图目前主要锁在 CAD/BIM 文件、Excel 表格和工程师经验里，缺少一个像 EDA 那样的**外部约束系统**。

---

## 二、HDA 与 EDA 的关系

HDA 不是替代 EDA，而是**把 EDA 的范式从电子系统扩展到所有物理硬件系统**。

| EDA（电子设计自动化） | HDA（硬件设计自动化） |
|---|---|
| 设计对象是芯片、PCB | 设计对象是机械、建筑、通信、能源等实体系统 |
| 真相源是 Verilog/VHDL/网表 | 真相源是 ADL（Assembly Definition Language）声明 |
| 验证层是 DRC、LVS、STA、SPICE | 验证层是 L0-L6 分层规则引擎 + CAD/CAE 后端 |
| 交付物是 GDS、网表、BOM | 交付物是 BOM、施工图、面板图、线缆表、仿真输入 |
| 核心问题是“晶体管怎么连” | 核心问题是“零件怎么存在、怎么配合、放在哪里” |

EDA 已经证明：**当设计意图变成结构化文本、规则变成可执行代码、变更进入版本控制，整个行业的知识再生产能力会被彻底重塑。** HDA 的目标是让实体工程也拥有同样的能力。

---

## 三、HDA 与 CAD/BIM 的关系

HDA 不是要消灭 CAD/BIM，而是要**重新定义它们在设计流程中的位置**。

| | 传统 CAD/BIM 范式 | HDA 范式 |
|---|---|---|
| 设计中心 | GUI 里的模型 | 文本声明（ADL） |
| 核心动作 | 点、拖、画 | 写声明、改参数、跑检查 |
| 真相源 | 专有模型文件 | YAML/TOML 文本 |
| 验证方式 | 人工审图 + 经验 | 规则引擎 + CAD/CAE 后端 |
| 变更管理 | 文件锁定、版本缀日期 | Git diff、PR review |
| AI/Agent 参与 | 困难（需 UI 自动化） | 原生（文本可直接读写） |

在 HDA 范式里，CAD/BIM 从“设计发生的场所”变成“被编排的验证与可视化后端”：

- **CAD 内核**负责精确几何检查：干涉、间隙、包容、可达性。
- **CAE 求解器**负责物理场仿真：CFD、FEA、电磁、振动。
- **可视化工具**负责给人看的 3D 预览、图纸、面板图。
- **HDA 层**负责统一入口：声明式建模、规则检查、版本控制、生成器编排。

这就像 Kubernetes 不替代容器，而是编排容器；HDA 不替代 CAD/CAE，而是编排它们。

## 四、HDA 与 SDH 的关系

**SDH 是思想，HDA 是赛道，piki 是参考实现。**

| | 定位 | 回答的问题 |
|---|---|---|
| **SDH（Software-Defined Hardware）** | 一种设计哲学 / 工程范式转移 | “工程设计应该怎么被组织？” |
| **HDA（Hardware Design Automation）** | 一个产品赛道或市场品类 | “面向非电子硬件的设计自动化工具应该长什么样？” |
| **piki** | SDH 框架的开源参考实现 | “我怎样用现成的工具构建一个 HDA 应用？” |

SDH 给出了五条核心原则：

1. **声明式**：用文本声明设计意图，而不是用鼠标拖拽。
2. **CLI 优先**：命令行是第一接口，Agent 和人类都能调用。
3. **开放数据格式**：YAML/Git/diff 是基础设施，专有格式是负债。
4. **Git / CI/CD 友好**：设计变更可 diff、可 review、可回滚。
5. **开源知识网络**：设计 know-how 以可执行约束的形式流通。

HDA 是这些原则在产品形态上的自然落地：一个 HDA 产品就是一套“文本声明 + 规则引擎 + 生成器 + 可视化消费层”的组合。

## 五、为什么 piki 让构建 HDA 应用变简单

在没有 piki 之前，如果你想为一个新领域（比如暖通、船舶、光伏阵列）做一个 HDA 工具，你需要从头做很多事情：

- 定义数据模型和 Schema
- 写 YAML/TOML 解析和源码位置追踪
- 设计引用完整性和 Mate 约束机制
- 实现规则引擎和诊断格式
- 实现生成器框架和报告输出
- 做 LSP 集成、CI 集成、Git 工作流

这些都不是领域知识，而是**基础设施重复建设**。

piki 把这些基础设施都做好了，并把它们抽象成可复用的层：

```text
┌─────────────────────────────────────────────┐
│           HDA 应用（你的领域）               │
│  插件：Family + Rule + Generator            │
│  项目：models/ + instances/ + layouts/      │
├─────────────────────────────────────────────┤
│              piki（SDH 编排框架）            │
│  插件发现 → ADL 加载 → 规则执行 → 生成器编排 │
├─────────────────────────────────────────────┤
│              adl（装配体定义语言）           │
│  PDL + PML + PLL 三子语言 + 统一诊断        │
└─────────────────────────────────────────────┘
```

你要做的只是把领域知识写成三样东西：

1. **Family**：这个领域有什么对象、有什么字段、约束是什么。
2. **Rule**：这个领域有哪些不能违反的硬约束。
3. **Generator**：这个领域需要输出什么交付物。

`piki init --plugin your-domain` 就能生成项目骨架，`piki check` 自动运行所有规则，`piki generate` 自动输出交付物。

### 例子：`samples/01-telecom-expansion`

`01-telecom-expansion` 就是 piki 之上一个典型的 HDA 应用雏形：

- **领域对象**：Rack、PDU、Server、Switch、Port、Transceiver、Fiber。
- **规则**：功率预算、U 位冲突、双路冗余、端口兼容、线缆长度、机柜承重、散热、维护空间。
- **生成器**：BOM CSV、机柜面板图、端口互连图、线缆标签、机房平面图、功率预算表。
- **验证后端**：目前是 AABB 快速几何和线性物理规则，未来可以对接 CAD 内核和 CFD。

它很粗糙，但它已经展示了 HDA 的核心 workflow：

```bash
edit instances/...yaml   # 改设计
piki check               # 自动校验
piki generate            # 生成交付物
git commit               # 版本化
```

## 六、HDA 的边界

HDA 不是“又一个试图包打一切的工业软件平台”。它有明确的边界：

**HDA 做：**

- 设计意图的声明式表达
- 确定性约束的自动检查
- 从声明到交付物的自动综合
- 版本控制、CI/CD、Agent 协作

**HDA 不做（或不是核心）：**

- 复杂几何建模内核（交给 CAD）
- 高保真物理仿真（交给 CAE）
- 制造执行/工艺控制（交给 MES/CAM）
- 项目管理/流程审批（交给 PLM/ERP）

HDA 是**设计阶段的编译器和综合器**，是 EDA 在更广阔实体工程领域的对应物。

---

## 七、总结

- **HDA** 清楚地表明了自己和 **EDA** 的承继关系，以及和 **CAD/BIM** 的范式差异。
- **SDH** 是 HDA 背后的思想：文本真相源、规则驱动、Git/CI 友好、Agent 可操作。
- **piki** 是 SDH 框架的参考实现，它让构建一个具体领域的 HDA 应用从“从头造轮子”变成“写插件 + 填数据”。
- `samples/01-telecom-expansion` 就是 piki 上 HDA 应用的一个早期粗糙样本。

如果你要向外解释 piki 是什么，最简单的一句话可以是：

> **piki 是一个开源的 SDH 框架，目标是让“为实体工程构建 HDA 应用”变得像为软件工程构建 DevOps 工具链一样简单。**
