# 软件定义工程（SDE）：大模型落地工业的必由之路

> 当前的大模型是“盲”的，当前的工业软件是“给健全人类用的”。当盲人试图指挥健全人操作复杂机器，摩擦与失败不可避免。解决之道不是让大模型视力更好，而是让工业软件变成可被非人类智能体直接操作的基础设施。

---

## 一、一场正在发生的错配

2024 年起，工业界对大模型的热情从“聊天”迅速转向“落地”。最常见的落地思路是：把大模型接到现有工业软件上，让它像人一样看界面、点按钮、填表单、出图纸。

这个思路很快遇到天花板：

- **LLM 是盲的**。它看不到屏幕，无法确认一个设备是否真的放在了正确的 U 位，无法判断线缆是否真的接对了端口。
- **VLM 是散光的**。它能截图、能识别 UI 元素，但定位不精确、对长程依赖理解差、对工程语义几乎无知。它可以把“插入光纤”理解为点击某个坐标，但无法验证这根光纤的速率、极性、长度是否匹配。
- **通过 UI 操作慢、不准确、难交验**。每个动作都要经过屏幕渲染、元素检测、坐标点击、结果观察，延迟高、鲁棒差、结果难以被其他系统复用。

与此同时，工业软件本身并没有为被机器操作而设计。它们的默认假设是：**使用者视力良好，受过专业训练，熟悉鼠标、菜单、对话框和快捷键**。这个假设在过去三十年是合理的，但在 Agent 时代变成了根本性的障碍。

这不是技术细节问题，而是**交互范式的不匹配**：一边是试图理解世界的多模态大模型，一边是为人类眼手协调优化的 GUI。让前者驱动后者，就像让一位盲人通过语音指挥一位健全人操作复杂机床——不是不可能，但成本高、错误多、难以规模化。

## 二、两个被忽视的结构性事实

### 事实一：大模型的“中期训练”正在代码领域发生

当前工业界讨论大模型时，经常强调“工业数据稀缺”。这个判断是对的，但只说对了一半。

更关键的事实是：**大模型的中期训练（mid training）正在以 Code RLVR 为主**。RLVR（Reinforcement Learning from Verifiable Rewards，基于可验证奖励的强化学习）已经在数学、代码等领域证明了规模化提升模型能力的路径。其前提是：存在大量**可验证、可自动评分**的任务。

代码之所以成为 RLVR 的主战场，不是因为代码数据多，而是因为代码拥有从语法到运行时反馈的完整可验证链条：

| 验证层级    | 代码领域       | 工程设计领域           |
| ----------- | -------------- | ---------------------- |
| L0 格式     | 语法高亮、缩进 | 文件格式合法性         |
| L1 Schema   | 类型系统       | Schema 约束            |
| L2 完整性   | 单元测试       | 引用完整性、接口兼容性 |
| L3 业务规则 | 集成测试       | 功率预算、空间冲突     |
| L4 运行时   | 端到端测试     | 几何碰撞、物理干涉     |
| L5 仿真     | 性能基准       | 热、流体、电磁仿真     |
| L6 评估     | 人工代码评审   | AI/专家综合评估        |

工程设计天然具备类似的结构：约束、规则、物理定律、制造规范、验收标准。但工业软件没有把这些结构暴露给模型。它们把工程语义埋在菜单、对话框、私有文件格式和云端数据库里。模型无法直接读取设计意图，无法直接获得可验证的奖励信号，因此无法在工程领域建立有效的 RLVR 飞轮。

**工业数据稀缺是结果，工业软件不可被机器操作才是原因。**

### 事实二：工业软件假设使用者是“视觉与操作能力正常的人类”

这是绝大多数工业软件在设计时隐含的假设：

| 假设                                     | 对 Agent 的影响                               |
| ---------------------------------------- | --------------------------------------------- |
| 用户能通过视觉快速理解复杂界面           | Agent 需要 VLM + OCR + 元素检测，成本高且易错 |
| 用户能根据经验判断哪些菜单里有需要的功能 | Agent 需要大量 domain-specific 的 UI 探索策略 |
| 用户能接受“在对话框里逐步输入”           | Agent 必须模拟鼠标键盘，动作序列冗长          |
| 用户会主动保存、命名、管理文件           | Agent 需要额外的状态追踪和恢复机制            |
| 用户能理解专有文件格式和二进制数据       | Agent 无法直接读取、diff、验证设计意图        |

这些假设在 GUI 时代是优势，在 Agent 时代是负债。它们让工业软件成为一座设计精美但只对人类开放的建筑——入口狭窄、台阶陡峭、没有盲道、没有电梯。

## 三、工业软件无障碍化：Headless 化运动

如果我们承认大模型（以及未来的工程 Agent）是重要的“新用户”，就必须重新审视工业软件的可访问性。

在消费电子领域，“无障碍化”意味着为视障、听障、行动不便者提供辅助功能。在工业软件领域，我们需要一场类似的**无障碍化运动**：不是为少数群体做适配，而是为所有非人类智能体开放操作接口。

我们把这场运动的核心称为 **Headless 化**。

“Headless”借用了无头浏览器（headless browser）的概念，但含义更深：

- **无强制 GUI**：软件的核心功能不依赖屏幕、窗口、对话框。
- **可脚本化**：所有操作都可以被程序化调用、编排、复现。
- **状态可导出**：设计数据和运行状态以开放格式暴露，可被外部系统读写。
- **行为可验证**：操作结果可以通过规则、测试、物理约束进行自动校验。

Headless 化不是“去掉 GUI”，而是**让 GUI 成为可选消费层，而不是唯一入口**。就像 Kubernetes 既有命令行也有 Dashboard，但真相源永远是结构化文本；工业软件也应该有强大的可视化界面，但其核心语义必须能被 Agent 直接读写。

---

## 四、从 Headless 到 SDE：软件定义工程

Headless 化只是第一步。更深层的变革是：让工程设计本身成为可被软件定义、可被机器协作的对象。

我们把这个趋势命名为 **SDE（Software Defined Engineering，软件定义工程）**。它不是简单的“用软件做工程”，而是：

> 工程设计的意图、约束、规则、流程和交付物，都以开放、声明式、可版本控制的形式存在，并可以被 Agent 和人类共同操作、校验、演进。

SDE 建立在四个“开放”之上：

### 1. 开放格式（Readable & Writable）

设计数据不锁在专有文件里，而以开放的文本格式存在。这意味着：

- Agent 可以直接读写设计意图
- Git 可以进行 diff、merge、blame
- 人类可以用任何文本编辑器审阅
- 规则引擎可以直接解析和校验

### 2. 开放接口（Orchestrable）

工业软件的每个核心操作都提供稳定的 API，可以被工作流引擎、CI/CD、Agent 编排。这意味着：

- 设计变更可以触发自动检查
- 规则违反可以阻断提交
- 交付物生成可以成为流水线的一步
- 多工具可以围绕同一数据源协作

### 3. 开放逻辑（Forkable & Contributable）

设计规则不是黑盒，而是可以被查看、修改、分叉、上游贡献的代码。这意味着：

- 企业可以把国标、企标、项目经验编码为规则
- 学术界可以把研究成果沉淀为可复用的验证器
- 社区可以共同审校和改进规则库
- 模型可以通过 RLVR 在规则环境中获得可验证奖励

### 4. 声明式建模 + Git Workflow + CI/CD

SDE 把工程设计与现代软件工程实践对齐：

| 软件工程 | SDE 对应                       |
| -------- | ------------------------------ |
| 源码文件 | 声明式工程文本                 |
| 编译器   | 规则引擎 + 校验器              |
| 单元测试 | 单条规则检查                   |
| 集成测试 | 跨记录业务规则 + 几何碰撞检查  |
| CI/CD    | 规则检查在提交前运行           |
| 版本控制 | Git diff / merge / review      |
| 制品     | 图纸、清单、报告、3D 场景      |

当工程设计可以被 `git diff`，可以被规则检查阻断，可以被 CI 编排，它就从一门手艺变成了一门可协作、可自动化、可规模化的工程学科。

## 五、piki：SDE 在实体工程中的实现

piki 的诞生正是为了推动 SDE 在实体工程领域落地。它是一个**声明式系统建模框架**，让 Agent 和人类能在同一份文本真相源上协作：选型、布局、集成、验证，并生成足以指导制造和安装环节的交付物。

piki 不是 CAD，不是仿真软件，不是 PLM。它定位在**设计意图层和规则校验层**：上游承接人类和 Agent 的意图，下游驱动 CAD/BIM、仿真、制造、安装等环节。

### 核心设计原则

#### 1. 文本是真相源，Agent 是第一操作者

设计数据以结构化文本存在，规则以可执行代码存在。Agent 直接读写文本，而不是模拟鼠标点击。人类通过自然语言或草图表达意图，Agent 将其转化为文本变更，规则引擎验证后入库。

#### 2. 意图与位置分离，方案比选 = Git 分支

设备“是什么”和“放哪里”被拆成独立层。同一台服务器在不同方案中可以被放到不同机柜，`git diff` 只显示位置变化，而不污染设备定义。SDE 中的设计探索由此获得版本控制原语。

#### 3. 关系与对象同等重要

piki 把工程中的关键关系都建模为一等公民：

- **Interface**：精确到端口（如服务器网口）；
- **Connection**：信号/能量链路（如光纤、电源线）；
- **Mating Graph**：物理配合关系（如导轨装配、电源接口配对）。

这让 Agent 在写设计声明时就能被阻止“把光纤插进电源口”，而不是等到 VLM 点击坐标后才发现。

#### 4. 多上下文：一张图里看到“我要设计什么”和“我必须服从什么”

既有设施、保密网络、其他标段、概念占位、自然环境——这些实体不是本次工程的内部资产，但又必须参与校验。piki 用 **Context** 统一建模它们的归属与可见性策略，让 Agent 在同一设计空间内同时处理内部设计与外部约束。

#### 5. 领域知识插件化、可分叉、可贡献

电信、数据中心、暖通、制造等领域的知识以独立规则包形式存在，通过标准包管理机制分发、在项目配置中启用。企业可以在上游规则包基础上分叉自己的企标规则，学术界可以把研究成果沉淀为可复用检查器。

#### 6. 从文本直接生成交付物与可视化

piki 从声明式设计派生物料清单、施工图、线缆表、3D 场景等制品；可视化工具提供 3D 预览。但可视化是消费层，不是设计层：看到问题后，修改必须回到文本或 Agent。

### 结果：设计成为可编译、可验证、可回滚的对象

在 piki 中，一次设计迭代看起来像软件工程：

```text
写设计声明 → 规则检查（多层自动校验） → 修正违规 → 生成交付物 → git commit
```

错误在提交前被发现，而不是在现场施工后。

## 六、SDE 的未来蓝图

piki 当前实现的只是 SDE 的第一块拼图。我们看到的未来包括以下方向：

### 1. 规则市场：从项目经验到行业知识

今天的规则散落在各个企业的私有仓库里。未来我们希望：

- 国标、行标以开源规则包形式维护；
- 设备厂商发布官方型号数据 + 配套验证规则；
- 设计院把项目特殊要求贡献回社区规则库；
- 学术界把研究成果沉淀为可复用的检查器。

规则库将成为工程领域的“开源依赖”，像今天开源依赖库支撑软件工程一样支撑 SDE。

### 2. 工程 Agent 的 RLVR 训练场

当足够多的设计数据以 Text-Native 形式存在，并且规则可以被自动执行时，工程设计就成为模型 RLVR 训练的理想领域：

- 模型提出一个设计方案（修改文本声明）；
- 系统在秒级返回规则检查反馈；
- 高成本仿真和专家评估为少数高质量样本提供最终奖励；
- 模型在数百万次自动迭代中学会多目标权衡、长程规划和物理一致性。

piki 的目标不是训练某个特定模型，而是提供这个训练环境本身。

### 3. 跨企业、跨仓库的设计协作

大型 EPC 项目涉及总包、分包、设计院、供应商。今天他们通过邮件传 Excel、图纸和 PDF。未来：

- 每个参与方维护自己的设计仓库；
- 通过标准引用机制互相引用资产与约束；
- Context 机制区分“我的设计范围”和“我引用的外部约束”；
- 接口变更通过 CI 自动触发依赖方规则检查，而不是在施工现场才发现冲突。

### 4. 从设计意图到制造安装的单向数据流

SDE 的终点不是图纸，而是让制造和安装环节直接消费结构化数据：

- 工厂从物料清单输出直接读取采购数据；
- 数控机床从开放几何格式读取加工数据；
- 施工队从施工图和线缆表按图施工；
- 运维系统从设计仓库继承资产台账和连接关系。

piki 作为设计意图层，成为物理世界数字孪生的上游真相源。

## 七、为什么是现在

SDE 之所以在这个时刻成立，是因为三个条件同时成熟：

1. **大模型能力达到临界点**：虽然不能可靠操作复杂 GUI，但已经能较好地读写结构化文本、解释规则、生成 diff。
2. **工程领域对自动化的需求迫近极限**：人力成本上升、项目复杂度上升、交付周期缩短，传统“人眼+鼠标”模式难以为继。
3. **开放生态成为竞争优势**：专有格式和封闭接口在 Agent 时代会加速贬值，能够被机器读写的开放系统将获得网络效应。

谁先完成 SDE 转型，谁就能让 AI 真正进入工程核心流程；谁固守 GUI-only 的旧范式，谁就会发现自己拥有世界上最精良的工具，却没有任何 Agent 能高效使用它。

---

## 八、结语：不是替代工程师，而是升级工程

SDE 不是要让 Agent 取代工程师。它的目标是：

> 让工程师从“画图+点击+校对”的执行者，变成“定义意图和规则、审阅 Agent 输出”的设计架构师。

人类仍然是不可替代的：复杂权衡、现场经验、价值判断、创造性设计。但这些高级能力只有在低级执行工作被有效自动化的前提下，才能得到最大程度的释放。

大模型是盲的，工业软件是给人看的。这个错配不会通过让大模型“视力更好”来解决——那太慢、太贵、太脆弱。正确的路径是：**让工业软件变得对机器友好，让工程设计变得像软件一样可定义、可校验、可协作**。

这就是 SDE，这就是 piki 想要推动的未来。
