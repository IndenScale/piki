# 软件定义硬件（SDH）：大模型落地工业的必由之路

> 当前的大模型是"盲"的，当前的工业软件是"给健全人类用的"。当盲人试图指挥健全人操作复杂机器，摩擦与失败不可避免。解决之道不是让大模型视力更好，而是让工业软件变成可被非人类智能体直接操作的基础设施。

---

## 一、一场正在发生的错配

2024 年起，工业界对大模型的热情从"聊天"迅速转向"落地"。最常见的落地思路是：把大模型接到现有工业软件上，让它像人一样看界面、点按钮、填表单、出图纸。

这个思路有两个层次的问题。一个是**暂时性的**，一个是**第一性的**。

### 暂时性问题：交互界面不匹配

LLM 看不到屏幕，VLM 定位不精确，通过 GUI 操作慢且易出错。工业软件的默认假设是"使用者有眼睛、有手、经过专业训练"——菜单、对话框、快捷键，一切为人类眼手协调优化。当 Agent 试图通过模拟鼠标键盘来操作这些界面时，效率低、鲁棒差。

这些问题可以——而且正在——被缓解。VLM 的视觉精度在提升。GUI 自动化框架在成熟。Agent 推理速度在加快。给它们几年时间，让 Agent 操作 CAD 界面也许不再是瓶颈。

**但这不是问题的根源。** 即使 Agent 能以 100% 精度操作任意 GUI，底层矛盾依然存在。

### 第一性问题：概率模型不承诺正确性

大模型的本质是**概率的**。它根据输入预测最可能的输出——但它在数学上没有"输出必须满足 N 个约束"的保证。一个 LLM 可以写出语法完美的代码，同时引入一个逻辑 bug。一个 VLM 可以精确点击屏幕上的每个按钮，但不知道点击顺序是否有物理意义。

软件工程容忍这一点，因为软件有**外部约束系统**：编译器检查语法，类型系统检查一致性，测试套件检查行为。Agent 写代码 → 编译/测试/CI 兜底。Agent 不需要永远正确，它只需要产出"大概率正确、错了能被秒级发现"的候选。

工程设计没有这个外部约束系统。一个机房的完整设计涉及数十条硬约束：

- PDU 总负载 ≤ 额定容量的 80%
- 同一机柜内设备 U 位不能重叠
- 光模块波长、速率、接头类型必须与两端端口精确匹配
- 设备深度不能超过机柜深度
- 核心设备必须双路供电

**每一条约束都是确定性的、可自动计算的——但当前的工业软件没有把它们暴露为可执行规则。** 它们锁在 CAD 文件里、藏在 Excel 公式里、存在工程师的直觉里。Agent 产出一个设计方案后，系统无法在秒级内回答"这个设计对吗？"

这就是错配的本质：**Agent 的底层数学决定了它必须配一个自动化的外部约束系统，而当前的工程工具链没有提供这个系统。**

概率模型不承诺正确性，这不可改变。但外部约束系统的缺失，是可以改变的。这就是 SDH 要解决的问题。

## 二、形式化程度光谱：工程设计的连续演进

上面的论述可能会造成一个误解：软件工程做对了，工业软件做错了——这是二元对立。

**真实情况不是二元，是连续谱。**

工程设计的形式化程度，从高到低是一条清晰的光谱：

```
高形式化 ←——————————————————————————————→ 低形式化

  数学        软件工程      IC/EE       机械/通信/建筑
  (纯理论)     (代码)      (EDA)        (CAD/BIM)
     │           │           │              │
  公理证明    编译+测试    DRC+LVS      人工校审+施工返工
  零歧义      确定性判对   规则自动判对   经验驱动
```

### 光谱左端：形式化的极致

**数学**是光谱的最左端。一个定理的证明是纯符号操作——要么对，要么错，没有中间态。所有推理步骤都可以被形式化验证（Coq、Lean）。

**软件工程**是数学向实践的第一个投影。源代码是文本，编译器和类型检查器是自动裁判，测试套件是行为契约。每一次提交都可以在秒级内获得确定性反馈：通过 or 不通过。

**IC/EE（芯片设计）**是软件工程向物理世界跨出的关键一步。EDA 工具链几乎完整复刻了软件工程的验证金字塔：Verilog 源码 → 逻辑综合 → 布局布线 → DRC（设计规则检查）→ LVS（网表一致性对比）→ 静态时序分析 → Signoff。每一步都有自动裁判。流片前的 Signoff 是极严酷的质量门——一条不通过，几亿打水漂。

**注意：EDA 不是"和 CAD 不同的另一类软件"。EDA 就是 CAD——它是形式化程度最高的制造业软件。** 芯片设计师用 EDA 做的事情，和机械工程师用 SolidWorks 做的事情，在物理本质上完全一样：把几何实体放在空间里，定义它们的配合关系和约束，验证它们不会互相干涉。区别只在于形式化程度：EDA 把每一步都做成了可自动验证的文本级操作，而 CAD 还停留在 GUI 交互。

### 光谱右端：尚未形式化的工程设计

机械设计、通信站点设计、建筑暖通设计——它们仍然以 GUI 为主要交互媒介。设计意图存在 DWG/RVT 文件里，验证靠人工逐项打勾，错误往往在施工阶段才发现。

它们和 EDA 的差距，不是领域复杂度不同，是**形式化程度不同**。而形式化程度，不是天生的——它是可以逐步提升的。

### 同一个方向，不同的速度

关键洞察在于：**SE 和 EDA 不是一开始就在光谱左侧的。它们在过去的 60 年里移动得更快。**

| 时代 | 软件工程 | IC/EE | 机械/通信工程 |
|------|---------|-------|------------|
| 1960s | 打孔卡，手工汇编 | 手画版图，用尺子量 DRC | 绘图板，丁字尺 |
| 1980s | C 编译器，make，lint | SPICE 仿真，早期 DRC | AutoCAD（数字图板）|
| 2000s | Git, CI/CD, 单元测试框架 | Verilog-AMS, UVM, STA | 3D CAD，PDM/PLM |
| 2020s | GitHub Copilot, RLVR, Agent | Agent 写网表，生成约束 | **仍在 GUI 层** |

软件工程从"手工打补丁"进化到"声明式基础设施 + Agent 自主修改"用了 60 年。EDA 从"手画版图"进化到"AI 生成网表过 DRC"用了 60 年。机械和通信设计——它们不是失败了，它们只是**才刚开始认真地往光谱左边移动**。

而 AI Agent 的出现，给这个移动加了一个从未有过的加速度。

### 形式化提升 = 帕累托改进

往光谱左边移动，不需要一步登天。每一个形式化步骤都是**帕累托改进**——它不会让任何现有能力变差，只会增加新的自动化收益：

- 把设备型号从 Excel 提取到结构化 YAML → **型号库可被 grep、diff、脚本消费**
- 把一条设计规范从 PDF 写成 Python 规则函数 → **每个项目自动检查，不再依赖人的记忆**
- 把"机柜深度匹配"从口头经验变成 Mate 约束 → **加载时自动校验，永久免疫此类错误**
- 把设计输出从手工画图变成生成器 → **BOM、面板图、端口图一键生成**

一个通信设计团队不需要变成芯片设计团队才能受益。**只要把今天最痛的那条检查——比如 PDU 功率预算——写成一条规则，明天就少返一个工。**

这就是 SDH 的核心主张：不是"全部或全不"，而是"每次向左移动一步，每一步都产生可测量的收益"。

## 三、为什么 SE 能喂养 RLVR 而工程领域不能

上面说的光谱，直接解释了一个具体的技术问题：为什么大模型的中期训练（RLVR）集中在代码领域，而工程领域吃不到？

### 事实一：大模型的"中期训练"正在代码领域发生

当前工业界讨论大模型时，经常强调"工业数据稀缺"。这个判断是对的，但只说对了一半。

更关键的事实是：**大模型的中期训练正在以 Code RLVR 为主**。RLVR（基于可验证奖励的强化学习）已经在数学、代码等领域证明了规模化提升模型能力的路径。其前提是：存在大量**可验证、可自动评分**的任务。

代码之所以成为 RLVR 的主战场，不是因为代码数据多，而是因为代码拥有从语法到运行时反馈的完整可验证链条：

| 验证层级    | 代码领域       | 工程设计领域           |
| ----------- | -------------- | ---------------------- |
| L0 格式     | 语法高亮、缩进 | 文件格式合法性         |
| L1 Schema   | 类型系统       | 字段约束与值域         |
| L2 完整性   | 单元测试       | 引用完整性、接口兼容性 |
| L3 业务规则 | 集成测试       | 功率预算、空间冲突     |
| L4 运行时   | 端到端测试     | 几何碰撞、物理干涉     |
| L5 仿真     | 性能基准       | 热、流体、电磁仿真     |
| L6 评估     | 人工代码评审   | AI/专家综合评估        |

工程设计天然具备类似的结构：约束、规则、物理定律、制造规范、验收标准。但工业软件没有把这些结构暴露给模型。它们把工程语义埋在菜单、对话框、私有文件格式和云端数据库里。模型无法直接读取设计意图，无法直接获得可验证的奖励信号，因此无法在工程领域建立有效的 RLVR 飞轮。

**工业数据稀缺是结果，工业软件不可被机器操作才是原因。**

### 事实二：代码拥有从语法到运行时的完整可验证链条——工程没有

这是绝大多数工业软件在设计时隐含的假设：

| 假设                                     | 对 Agent 的影响                               |
| ---------------------------------------- | --------------------------------------------- |
| 用户能通过视觉快速理解复杂界面           | Agent 需要 VLM + OCR + 元素检测，成本高且易错 |
| 用户能根据经验判断哪些菜单里有需要的功能 | Agent 需要大量领域特定的 UI 探索策略          |
| 用户能接受"在对话框里逐步输入"           | Agent 必须模拟鼠标键盘，动作序列冗长          |
| 用户会主动保存、命名、管理文件           | Agent 需要额外的状态追踪和恢复机制            |
| 用户能理解专有文件格式和二进制数据       | Agent 无法直接读取、diff、验证设计意图        |

这些假设在 GUI 时代是优势，在 Agent 时代是负债。它们让工业软件成为一座设计精美但只对人类开放的建筑——入口狭窄、台阶陡峭、没有盲道、没有电梯。

## 四、工业软件无障碍化：Headless Engineering

面对这个错配，最常见的补救思路是给工业软件加 API 包装层——用 RPA、UI 自动化、插件系统让 Agent 间接操作旧软件。

这实际上是在"盲人指挥健全人"的模型上打补丁。它有几个致命缺陷：

- **语义层仍然是 GUI**。API 包装层传递的是 UI 事件（点击、输入、拖拽），而不是设计意图（"SRV-01 安装在 RACK-A01 的 U18-19，功率分配来自 PDU-A 端口 3"）。
- **验证代价昂贵**。每次操作后要截图确认、OCR 识别、状态推断——成本高、可靠性差。
- **不可组合**。多个 Agent 不能同时操作一个界面，两个设计变更不能通过 git merge 自动合并。
- **不可版本化**。旧软件里的"版本历史"是操作序列的录像，不是设计意图的 diff。

正确的路径是**工业软件无障碍化**——不是给旧软件加辅助功能，而是重新设计底层，使设计意图以可被机器直接操作的形式存在。我们称之为 **Headless Engineering**：

1. **设计意图以文本声明形式存在**。用人类可读、机器可解析的声明式语言描述"什么东西存在、放在哪里、怎么连接"，而不是用点击和拖拽。
2. **规则引擎提供秒级、确定性、可解释的验证反馈**。Agent 提交修改后，几百毫秒内就能知道是否符合约束——不是一个需要人类用眼睛确认的可视化结果，而是一组结构化的 pass/fail 诊断。
3. **生成器将文本声明转化为交付物**。BOM 清单、配电图、施工图、电缆表等交付物不再是人工绘制的结果，而是从文本声明出发的可重复生成产物。
4. **可视化是消费层，不是设计层**。3D 预览、AR 叠加、图纸等人类需要的视觉输出仍然存在，但它们是只读的——看到问题后，修改必须回到文本或 Agent。

## 五、SDH 的核心类比：工程设计 = 软件工程

在 SDH 的框架下，一次设计迭代看起来像软件工程：

```text
写设计声明 → 规则检查（多层自动校验） → 修正违规 → 生成交付物 → git commit
```

这不是隐喻，而是真实的 workflow。工程师（或 Agent）用声明式语言定义设备、布局、连接和配合。规则引擎在秒级内从语法到空间碰撞逐层检查。诊断信息精确到行号，可以逐条修复。交付物从声明自动生成，而非人工绘制。文本文件天然支持 diff、review、merge、blame。

错误在提交前被发现，而不是在现场施工后。

这就是 SDH 的本质：**不是用软件辅助人来设计工程，而是让工程设计本身像软件一样被定义、被验证、被版本化、被自动化。**

## 六、SDH 的未来蓝图

### 1. 规则市场：从项目经验到行业知识

- 国标、行标以开源规则包形式维护
- 设备厂商发布官方型号数据与配套验证规则
- 设计院把项目特殊要求贡献回社区规则库
- 学术界把研究成果沉淀为可复用的检查器

### 2. 工程 Agent 的 RLVR 训练场

- 模型提出设计方案（修改文本声明）
- 系统在秒级返回规则检查反馈
- 高成本仿真和专家评估为少数高质量样本提供最终奖励
- 模型在数百万次自动迭代中学会多目标权衡、长程规划和物理一致性

### 3. 跨企业、跨仓库的设计协作

- 每个参与方维护自己的设计仓库
- 通过标准引用机制互相引用资产与约束
- 区分"我的设计范围"和"我引用的外部约束"
- 接口变更通过 CI 自动触发依赖方规则检查

### 4. 制造端的非确定性

一个诚实的限定：SDH 的"编译"类比不能无限延伸。

软件编译是确定性的——同样的源代码产出的字节码完全一致。工程设计的"编译器"却是供应链、非标物料、车间机床和现场施工人员。文本声明中一个完美的 `rack-mount-19inch` 配合约束，在现场可能因为钣金件 2mm 的加工形变而完全失效。公差、良率、材料批次差异——这些物理世界的噪声，是当前声明式文本难以完全建模的。

这不是否定 SDH，而是划定了它的边界：SDH 保证的是**设计意图的正确性**，不是**物理实现的完美性**。前者是必要条件，后者需要制造端的标准化配合。

这自然引出一个更大的命题——**制造即服务（Manufacturing as a Service, MaaS）**：当各个加工厂乃至加工机器人能够接收标准的工艺文件并确定性执行加工时，SDH 的"编译"类比才能真正闭环。从设计声明到工艺文件到加工执行，形成一条完整且可验证的单向数据流。这是 SDH 的下一个前沿。

### 5. 从设计意图到制造安装的单向数据流

- 工厂从结构化 BOM 直接读取采购数据
- 数控机床从开放几何格式读取加工数据
- 施工队从施工图和线缆表按图施工
- 运维系统从设计仓库继承资产台账和连接关系

## 七、为什么是现在

SDH 之所以在这个时刻成立，是因为三个条件同时成熟：

1. **大模型能力达到临界点**：虽然不能可靠操作复杂 GUI，但已经能较好地读写结构化文本、解释规则、生成 diff。
2. **工程领域对自动化的需求迫近极限**：人力成本上升、项目复杂度上升、交付周期缩短，传统"人眼+鼠标"模式难以为继。
3. **开放生态成为竞争优势**：专有格式和封闭接口在 Agent 时代会加速贬值，能够被机器读写的开放系统将获得网络效应。

谁先完成 SDH 转型，谁就能让 AI 真正进入工程核心流程；谁固守 GUI-only 的旧范式，谁就会发现自己拥有世界上最精良的工具，却没有任何 Agent 能高效使用它。

---

## 八、结语：不是替代工程师，而是升级工程

SDH 不是要让 Agent 取代工程师。它的目标是：

> 让工程师从"画图+点击+校对"的执行者，变成"定义意图和规则、审阅 Agent 输出"的设计架构师。

人类仍然是不可替代的：复杂权衡、现场经验、价值判断、创造性设计。但这些高级能力只有在低级执行工作被有效自动化的前提下，才能得到最大程度的释放。

大模型是盲的，工业软件是给人看的。这个错配不会通过让大模型"视力更好"来解决——那太慢、太贵、太脆弱。正确的路径是：**让工业软件变得对机器友好，让工程设计变得像软件一样可定义、可校验、可协作**。

这就是 SDH——不是用软件辅助工程设计，而是用软件工程的方法论定义和管理硬件。
