# 开源工业软件 2.0：从工具到 know-how 流通基础设施

> 一个老工程师退休时，他带走的不是代码，而是那些从未被写下来的判断。一次型号归零后，整改报告被锁进柜子，下一个型号从零开始重新学习。这不是某个企业的问题，而是整个工业软件行业的结构性缺陷：**工具越来越强大，知识却越来越封闭。**

开源工业软件 2.0 要解决的不是“有没有更好的 CAD 或仿真器”，而是**工业 know-how 如何像代码一样被保存、被流通、被继承**。

---

## 一、一个工程师的退休

想象这样一个场景：

某航空型号项目进入收尾阶段。总师老张即将退休。过去三十年，他经手过七次类似的归零：某型接插件在高原低温下接触不良，某型电缆在振动试验中磨破绝缘层，某型液冷接头在长期使用后密封老化。每一次，团队都写了归零报告，改了设计，做了验证。

但老张退休后，这些知识去哪里了？

- 一部分在归零报告的 PDF 里，躺在档案柜或共享盘深处；
- 一部分在老张的脑子里，跟着他一起离开了；
- 一部分变成了年轻工程师的“经验”，但只在他们经手类似项目时才被调用；
- 极少一部分沉淀为企业标准，但标准更新慢，且往往是自然语言描述，难以直接指导设计。

下一个型号开始时，新的设计团队翻开型号任务书，面对的是几乎空白的历史。他们可能会重犯老张年轻时犯过的错误，然后再次归零、再次整改、再次锁柜。

这不是效率问题，而是**知识再生产问题**。

---

## 二、工业软件的两种历史

工业软件行业过去三十年主要解决的是**工具问题**：

- CAD 让几何建模更快；
- CAE 让仿真更精确；
- PLM 让数据管理更系统；
- EDA 让芯片设计更自动化。

这些工具极大地提升了单点效率，但它们默认用户是“有经验的人类工程师”。工具本身不保存经验，也不流通经验。一个接插件在高原低温下失效的教训，不会自动变成下一个设计工具里的约束。

开源软件行业则主要解决的是**知识流通问题**：

- 一个 bug 被发现；
- 有人提交 issue；
- 社区定位根因；
- 有人写 patch；
- patch 被 review 后合并到上游；
- 所有下游用户自动获得修复。

在开源软件里，**一次失败可以立即成为全球公共品**。但在工业软件里，一次失败往往只成为一个项目的内部教训。

这就是两种历史的分野：

| | 传统工业软件 | 开源软件 |
|---|---|---|
| 核心资产 | 许可证 + 专有文件格式 | 源代码 + 社区知识 |
| 失败处理方式 | 项目内归零、整改、归档 | issue → PR → 上游 → 生态 |
| 知识保存 | 文档、经验、标准 | 可执行代码、测试、约束 |
| 知识流通 | 人际传递、会议、培训 | git diff、PR review、版本升级 |
| 更新方式 | 等厂商发版 | 社区持续迭代 |

工业软件需要向开源软件学习的地方，不是“把软件开源”，而是**把 know-how 变成可流通、可验证、可继承的数字资产**。

## 三、传统闭环为什么断裂

传统工业界并非不重视知识管理。归零、复盘、FMEA、设计准则、企业标准——这些机制都在试图保存 know-how。但它们的闭环在几个关键环节断裂了：

### 1. 从“整改”到“约束”的距离太远

归零报告通常描述的是：

> “某型接插件在振动环境下松脱，建议改用带锁扣结构。”

这是一种**建议**，不是**可执行约束**。下一个设计师读到这份报告时，需要自行判断：我的设计是否属于“振动环境”？我的接插件是否有锁扣？锁扣是否满足要求？

如果这份知识被写成：

```yaml
# mates/connector/vibration-lock.yaml
type: vibration-resistant-connector
parent: PANEL-01
child: CONN-01
constrains:
  - field: locking_mechanism
    operator: "=="
    value_ref: "self-locking"
    message: "振动环境下接插件必须采用自锁结构"
```

那么下一次设计时，引擎加载就会直接报错，而不是等试验后才发现。

### 2. 知识被锁定在人和文档里

文档和经验的最大问题是：**它们不能被机器直接执行**。

一个老工程师知道“这个位置的电缆要留 20mm 余量”，但这种知识无法被 CAD 自动检查，无法被 Agent 读取，也无法跨项目自动继承。它只能依赖人。

### 3. 跨组织流通成本极高

即使企业内部有知识库，供应商、合作伙伴、学术界通常无法访问。一个整机厂发现的失效模式，供应商不知道；一个高校研究出的新约束，企业标准更新慢。

知识的流通被组织边界、文件格式、商业机密和人际关系层层阻断。

### 4. 传统商业软件的流通速度太慢

传统工业软件不仅不促进 know-how 流通，其商业模式反而在多个层面加速知识的封闭：

- **本地化部署，数据回传管道不畅**

  大型工业软件通常部署在企业内网或私有云，仿真数据、失效数据、设计变更难以回流到软件厂商，更不可能回流到行业公共知识库。一个客户发现的失效模式，无法被产品快速吸收并分发给其他客户。

- **许可证模式把大量用户排除在外**

  工业软件按席位、按模块、按核时收费，导致只有核心团队能使用。供应商、小公司、高校、独立工程师被挡在门外。他们既是知识的潜在贡献者，也是潜在受益者。许可证墙让知识只能在小圈子里流动。

- **插件定制化导致碎片化生态**

  商业软件厂商为了满足不同客户需求，往往做大量定制开发和私有插件。这些插件互不兼容、无法共享，形成无数个信息孤岛。同一个行业的约束，被重复实现了几十次，每次都在企业内部封闭循环。

## 四、开源工业软件 2.0 的四个条件

要让工业 know-how 像代码一样流通，需要四个基础设施条件：

### 1. Text-Native：机器可读

设计意图必须首先是文本，而不是专有格式或图形界面状态。文本可以被：

- Agent 直接读写；
- Git diff 和 merge；
- 规则引擎解析；
- CI 系统批量检查。

piki 选择 YAML 作为具体语法，不是因为 YAML 完美，而是因为它**同时对人可读、对机器可解析**。

### 2. Git-Native：可版本化

知识的演进历史必须被保留。谁、在什么时间、为什么增加了一条约束？这条约束来自哪个型号的哪次归零？Git 提供了这种可追溯性。

在 piki 中，一条新约束的提交记录可能是：

```
commit a1b2c3d
Author: Li Wei <liwei@example.com>
Date:   2026-06-14

    增加高原低温接插件约束

    来源：XX-2026 型号高原试验归零报告 R-026
    根因：某型接插件在 -40°C 下接触电阻增大
    措施：低温环境必须使用镀金厚度 ≥ 1.27μm 的接插件
```

### 3. Rule-Native：可执行

知识必须能被自动验证。自然语言标准是人类共识，但机器无法执行。只有把标准转化为规则、约束、Mating，才能实现“加载即检查”。

piki 的 `@rule` 和 `Mate.constrains` 就是为了让 know-how 从“文档”变成“可执行约束”。

### 4. Open-Source：可跨组织继承

知识的最大价值在于流通。当一条约束被提交到开源插件上游，所有下游项目、所有企业、所有型号都能自动继承。

这不是乌托邦。开源软件已经证明：**当知识的边际流通成本趋近于零时，整个生态的进化速度会指数级提升。**

## 五、piki：这个范式的基础设施

piki 不是开源工业软件 2.0 的全部，但它是关键基础设施之一。它提供了 know-how 流通所需的核心抽象：

| piki 概念 | 在 know-how 流通中的作用 |
|---|---|
| **PDL / Family / Model / Catalog** | 定义“部件是什么”，把产品规格机器可读化 |
| **PLL / Layout** | 定义“部件在哪里”，让部署方案可 diff、可分支 |
| **PML / Mating** | 定义“部件如何耦合”，把工程约束沉淀为可执行结构 |
| **@rule** | 把经验判断编码为自动检查 |
| **插件** | 让领域 know-how 以独立包形式分发和继承 |
| **Catalog** | 把真实世界产品映射到规则库 |
| **Context** | 标记既有/失效/参考实体，让历史参与校验 |
| **Generator** | 从文本真相源直接生成交付物 |

这些抽象共同构成一个闭环：

```
失效发生
   ↓
在 piki 中定位根因（LSP Diagnostic）
   ↓
总结失效模式，写成 @rule 或 Mate type
   ↓
提交到领域插件上游
   ↓
所有下游项目自动继承新约束
   ↓
新型号设计时直接面对规则库
   ↓
失败不再重复发生
```

## 六、约束即需求：新型号如何被规则库反向驱动

传统产品研发是“设计 → 验证 → 修改”的循环。开源工业软件 2.0 把这个循环倒过来：

> **规则库成为设计输入，产品在设计空间中被约束“生长”出来。**

想象一个连接器厂商要设计一款新型号。在传统流程中，他们需要：

1. 市场调研；
2. 阅读各行业标准；
3. 咨询老客户；
4. 设计原型；
5. 做大量试验验证兼容性。

在 piki 生态中，他们可以：

1. 拉取电信、数据中心、汽车等目标行业的 piki 插件；
2. 查看这些行业已有的 Mate type 和默认约束；
3. 把新连接器设计为一个 `Model`；
4. 用 `piki check` 验证它能否通过目标行业的规则库；
5. 通过验证后，把新连接器作为 `Catalog` 条目贡献回上游。

这意味着：

- **市场准入门槛降低**：新产品只需通过规则库即可被行业接受；
- **试错成本降低**：不需要为每个客户单独做全套兼容性试验；
- **创新方向更明确**：规则库反向告诉设计师，行业真正需要什么样的兼容性。

这就是“约束即需求”（Constraints as Requirements）：

> **行业规则库不是事后检查清单，而是产品设计的先验条件。**

## 七、公共知识与商业机密：如何分层开放

讲到“开源”和“跨企业共享”，一个立刻会浮现的问题是：

> “供应商和主机厂难道愿意让竞争对手受益吗？商业机密怎么办？”

这是一个真实且重要的问题。答案是：**不是把所有知识都开源，而是把知识分层，让不同层级以不同方式流通。**

### 知识的三个层级

| 层级 | 内容 | 开放性 | 例子 |
|---|---|---|---|
| **L1：物理定律与工程约束** | 不依赖具体产品的通用知识 | 适合完全开源 | “振动环境下自锁接插件更可靠”、“质量守恒要求入口流量等于出口流量” |
| **L2：行业惯例与标准约束** | 特定行业的通用要求 | 行业协会或联盟维护 | 航空接插件的盐雾试验要求、数据中心 PDU 的功率预算规则 |
| **L3：企业专有知识** | 与具体产品、工艺、供应链相关 | 企业私有 | 某型飞机的线缆走向、某厂商的独家工艺参数、具体供应商的报价与交付周期 |

开源工业软件 2.0 的目标是让 **L1 和 L2 尽可能流通**，而 **L3 留在企业内部**。

### 为什么企业愿意贡献 L1/L2？

1. **L1 不是竞争优势**

   物理定律和通用工程约束不是任何企业的护城河。把这些开源，不会削弱竞争力，反而能减少全行业的重复试错成本。

2. **L2 的开源降低合规成本**

   如果所有企业都按各自理解实现行业标准，会导致无数碎片化规则库。共建一个开源的行业规则库，反而让所有人省下重复实现和解释标准的成本。

3. **贡献者获得话语权**

   在开源规则库中贡献约束的企业，实际上参与了行业标准的形成。这和参与 ISO、IEC 标准制定类似，但速度更快、执行更直接。

4. **供应商有动力让自己的产品被规则库接纳**

   连接器厂商希望自家型号出现在规则库的推荐列表中；芯片厂商希望自家接口被规则库支持。规则库成为事实上的市场准入通道。

### piki 如何支持分层开放？

piki 的插件架构天然支持这种分层：

- **上游开源插件**：承载 L1 和 L2，如 `piki-telecom`、`piki-datacenter`、`piki-fluid`；
- **企业私有插件 / 规则包**：承载 L3，如 `piki-boeing-internal`、`piki-huawei-enterprise`；
- **Catalog 分级**：公共 Catalog 共享标准件信息，私有 Catalog 保存企业供应链数据；
- **Context 标记**：用 `context: classified` 或 `context: external` 控制可见性，确保机密实体只参与必要检查，不进入公开报告。

```yaml
# 公共规则库（上游开源）
# mates/connector/vibration-lock.yaml
type: vibration-resistant-connector
constrains:
  - field: locking_mechanism
    operator: "=="
    value_ref: "self-locking"
```

```yaml
# 企业私有规则包
# .piki-enterprise/boeing/wiring-harness-rules.yaml
rule_id: BOEING-WIRE-001
scope: classified
message: "某型机特定线束余量要求"
```

### 竞争企业如何共享基础约束？

关键在于**共享约束，不共享设计**。

竞争企业可以共同维护：

- “接插件在振动环境下必须自锁”——这是通用约束；
- “某温度区间内必须使用某等级绝缘材料”——这是行业约束。

但各自保留：

- 具体使用哪个供应商的接插件；
- 整机内部的具体布局和线缆走向；
- 与性能和成本相关的专有参数。

这就像汽车厂商共同维护 OBD 协议，但各自保留发动机和底盘的详细设计。共享的是接口语言和最低约束，不是产品本身。

## 八、未来图景：一个失效，整个行业更强

在开源工业软件 2.0 的愿景中，一次工程失效的处理方式会完全不同：

### 今天

```
某型号接插件在高原低温下失效
        ↓
项目团队归零、整改
        ↓
写一份报告，锁进柜子
        ↓
老张退休，知识流失
        ↓
三年后另一个型号重蹈覆辙
```

### 未来

```
某型号接插件在高原低温下失效
        ↓
项目团队用 piki 定位根因
        ↓
写成 Mate type：high-altitude-low-temp-connector
        ↓
提交到 telecom 插件上游
        ↓
所有使用该插件的项目自动获得约束
        ↓
连接器厂商看到规则库，设计更耐低温的型号
        ↓
新型号通过规则库验证，直接获得市场准入
        ↓
整个行业的高温/低温接插件设计水平提升
```

这不是单个工具的改进，而是**工业知识再生产方式的范式转变**。

---

## 九、这不仅关乎 piki

开源工业软件 2.0 需要的不只是 piki。它需要：

- **开放的规则库**：像 Maven/npm 一样分发领域约束；
- **可验证的 benchmark**：如 [SD-HWE-Bench](../initiatives/001-sd-hwe-bench.md)，让 Agent 有训练目标；
- **开放的 CAD/CAE 接口**：让设计工具可以被程序化调用；
- **开放的材料与元器件数据**：让 Catalog 成为真实世界的机器可读映射；
- **跨企业的协作协议**：让竞争企业也能共享基础约束而不泄露核心机密。

piki 的价值在于：它提供了一种**文本原生的、可版本化的、可执行的 know-how 表达协议**。在这个协议之上，可以生长出整个开源工业软件生态。

---

## 十、结论

传统工业软件让我们能更快、更精确地设计单个产品。但它没有解决一个更根本的问题：**设计知识如何在组织之间、型号之间、代际之间流通。**

开源工业软件 2.0 的核心命题是：

> **让工程 know-how 像开源代码一样被保存、被流通、被继承。**

piki 是这个命题的一个具体实现。它通过 Text-Native、Git-Native、Rule-Native、Open-Source 四个条件，把原本锁在柜子里的归零报告和工程师脑子里的经验，转化为可执行的约束和可共享的规则库。

最终目标不是取代 CAD、仿真或 PLM，而是在它们之上建立一层**设计意图与 know-how 的流通基础设施**：

- 一个失效，可以被整个行业学习；
- 一条约束，可以成为新型号的设计输入；
- 一个插件，可以让所有下游项目自动变强。

这才是开源工业软件的未来。
