# Engineering RLVR：可验证奖励如何驱动工程 AI

> AlphaGo 赢在围棋，DeepSeek-R1 突破数学推理，Claude 在 SWE-Bench 上超越人类——这些里程碑背后是同一种驱动力：可验证奖励（Verifiable Reward）。Game → Math → Code → HWE。硬件工程（Hardware Engineering）是 RLVR 的下一个主战场——不是因为它像代码，而是因为它的可验证约束密度比代码更高。

---

## 一、RLVR 的扩张路径

RLVR（Reinforcement Learning from Verifiable Rewards）的核心思想极其简单：如果一个问题有客观的、可自动计算的正确性判定，模型就可以通过大规模试错来学会解决它——不需要人工标注，不需要人类偏好数据。

这个范式在过去十年间沿着一条清晰的路径扩张：

| 里程碑 | 领域 | 验证方式 | 反馈延迟 |
|---|---|---|---|
| AlphaGo (2016) | 围棋 | 胜负规则（二值） | 终局 |
| DeepSeek-R1 (2025) | 数学 | 最终答案正确性 + GRPO 组内排名 | 秒级 |
| Claude Code (2025) | 软件工程 | 编译 + 测试套件 + 类型检查 | 秒-分钟 |
| **HWE（目标）** | **硬件工程** | **分层规则引擎（L0-L6）** | **毫秒-分钟** |

共同内核：**可验证奖励是唯一燃料。** 哪个领域先建立起廉价、确定性、可自动计算的验证系统，哪个领域就能率先进入 RLVR 飞轮。

---

## 二、HWE 的燃料密度优势

硬件工程有一个被低估的优势：**约束密度极高，且约束与设计质量天然高相关。**

一个电信机房的端到端设计同时涉及：

- 功率预算不等式（每台设备 TDP 求和 ≤ PDU 容量的 80%）
- U 位容量约束（每机柜 42U，设备高度累积不能超过）
- 接口兼容性矩阵（SFP28 端口只能接 SFP+/SFP28 模块，不能接 QSFP28）
- 信号拓扑完整性（每台服务器必须有冗余上行链路）
- 安装工法前提（特定设备需要特定工具和空间余量）

**每个约束都是确定性的、秒级可评估的。** 物理定律不关心设计风格。一个违反功率预算的设计在任何评审标准下都是错的，不存在"主观评分类"的模糊地带。

这恰恰是 RLVR 的理想条件：清晰的对错边界、廉价的验证、高容量的训练信号。

## 三、四个基础设施缺口

尽管 Engineering RLVR 在理论上有极高的燃料密度，但实际启动面临四个基础设施缺口：

### 缺口一：自包含（Self-Contained）

软件工程中，`git clone` 一个仓库就能得到项目的一切：代码、配置、测试、依赖声明。工程领域完全相反——真相源散落在邮箱附件、IM 对话、厂商网站、老师傅记忆、内部 SharePoint 里。

**需要什么**：所有设计意图以文本形式存在于单一仓库中。`git clone` + 一条命令即可复现完整的验证环境。

### 缺口二：可操作（Operable）

LLM 操作代码是文本原生的——grep、diff、git log 都是标准工具。操作 CAD 文件需要通过 GUI 过程式操作——每个动作都要经过屏幕渲染、元素检测、坐标点击。

**需要什么**：文本声明是唯一真相源。Agent 直接读写结构化文本，通过命令行获得结构化验证反馈。

### 缺口三：自动化验证（Automated Verification）

有限元仿真太慢（小时级），CAD 碰撞检测太脆弱且依赖 GUI。需要秒级、确定性、可解释的轻量验证层——规则引擎，不是仿真求解器。

**需要什么**：分层规则引擎，从语法到空间碰撞全部在毫秒到秒级内返回结构化诊断。

| 层级 | 检查内容 | 反馈延迟 | 等价软件概念 |
|---|---|---|---|
| L0 | 文件格式合法性 | < 1ms | 语法解析 |
| L1 | Schema 校验（类型、值域） | < 10ms | 类型检查/linter |
| L2 | 引用完整性 | < 10ms | 链接检查 |
| L3 | 业务规则（功率、U 位、接口兼容） | < 100ms | 单元测试 |
| L4 | 几何检查（碰撞检测） | < 1s | 集成测试 |
| L5 | 仿真校验（CFD/FEA，可选） | 分钟级 | E2E 测试 |
| L6 | 下游工具链生成 | 分钟级 | 部署验证 |

### 缺口四：清晰任务定义（Clear Task Definition）

软件工程有 GitHub issue——问题描述 + 上下文 + 预期行为 + 复现步骤，纯文本，一个 URL。工程领域没有等价物。

**需要什么**：每个任务是从自然语言需求到可评分设计产出的自包含单元。Agent 从 NL 需求出发，产出结构化设计声明，评分由规则引擎自动完成。

## 四、SD-HWE-Bench：Engineering RLVR 的竞技场

SD-HWE-Bench（Software-Defined Hardware Engineering Benchmark）是填补上述四个缺口的尝试——一个标准化的、开放的、可自动评分的评测基准，驱动 Agent 在工程设计领域的 RLVR 训练。

### 核心范式

```
Input:  自然语言工程需求
Output: 结构化设计声明
Score:  L0-L4 规则检查 Pass@k + 交付物生成质量
```

### 任务分类

六种原子能力，覆盖从简单到复杂的工程推理：

| 任务类型 | 考察能力 | 难度 |
|---|---|---|
| instance-declaration | 型号选择、参数赋值、属性推导 | Easy |
| layout-design | 离散/连续空间分配、约束满足 | Easy |
| connection-design | 接口兼容匹配、拓扑推理 | Medium |
| mating-design | 机械约束匹配、耦合类型选择 | Medium |
| comprehensive | 全链路协调、多约束并行满足 | Hard |
| incremental | 变更影响分析、约束保持 | Medium（规划中） |

每类任务还有三档难度：L1 直接（需求给出具体参数）→ L2 推理（需求给约束条件）→ L3 规划（需求只给高层目标）。

### 当前基线数据（2026-06-14）

3 个模型 × 5 个任务：

| Task | DeepSeek V4 Flash | DeepSeek V4 Pro | Kimi K2.7 Code |
|---|---|---|---|
| comprehensive-001 | ❌ 20% | ❌ 20% | ❌ 85% |
| connection-design-001 | ❌ 60% | ❌ 60% | ❌ 85% |
| instance-declare-001 | ❌ 20% | ❌ 30% | ⏳ |
| layout-design-001 | ❌ 30% | ❌ 65% | ❌ 0% |
| mating-design-001 | ✅ 85% | ✅ 85% | ❌ 85% |
| **Pass@1** | **1/5 (20%)** | **1/5 (20%)** | **0/4 (0%)** |
| **Avg Score** | **43%** | **52%** | **64%** |

> Pass@1 定义：overall_score ≥ 0.6 视为通过。

**三个关键发现：**

1. **交付物生成是主要瓶颈**。模型能通过 L1-L4 规则校验（功率预算、U 位碰撞、接口兼容），但无法生成 BOM CSV、power budget 等结构化交付物。Kimi 在三个子任务均得 85%，但因交付物失败导致 Pass@1 = 0/4。

2. **端到端任务仍然极难**。在唯一真正的端到端任务（complete 42U 机柜从零部署）上，DeepSeek V4 Flash 和 V4 Pro 均只得 20%——卡在 schema 验证、引用完整性和业务规则。

3. **最好的模型也缺乏工程常识**。三个模型平均得分 43%-64%，没有一个达到 Pass@1 的 20% 以上。这证实了：即使强 LLM 在结构化工程设计任务上也存在根本性的能力缺口。

## 五、与 SWE-Bench 的差异

| 维度 | SWE-Bench | SD-HWE-Bench |
|---|---|---|
| 输入 | GitHub issue（已有上下文） | 自然语言工程需求 |
| 输出 | 代码 patch（修改已有代码库） | 声明式工程文本（从零创建设计） |
| 验证 | 单元/集成测试 | 分层规则引擎 + 交付物生成检查 |
| 正确性 | 多为二值 | 支持 partial credit |
| 领域成熟度 | 成熟（数千个任务） | 早期（5 个任务，目标 30+） |

SD-HWE-Bench 继承了"真实任务抽取 + 自动评分 + Pass@k"的方法论，但面对的是完全不同的领域挑战：从零创建设计而非修改已有代码、多层级协作而非单文件 patch、确定性规则评分而非依赖测试套件。

## 七、Engineering RLVR 的战略意义

Engineering RLVR 定义了这个生态的**进化引擎**：

```
Benchmark 定义任务和评分
       ↓
Agent 在 Benchmark 上训练
       ↓
Agent 产出更好的设计
       ↓
发现新的失效模式
       ↓
失效沉淀为规则 / 配合约束
       ↓
约束库让 Benchmark 更难
       ↓
Agent 必须更强才能通过
       ↓
（循环）
```

这是一个自我强化的飞轮。每一次失败都让约束库更完善；每一条新规则都让 Benchmark 更有区分度；每一次 Agent 能力的提升都让工程自动化更近一步。

---

## 八、与现有工作的关系

- **AMS-IO-Bench**（芯片 I/O 环设计）：最接近的"存在性证明"——70%+ DRC+LVS 通过率证明了"结构化设计意图 → 规则验证"范式在工业工作流中可行。SD-HWE-Bench 将它推广到系统级硬件工程。

- **AEC-Bench**：互补关系。AEC-Bench 评估 AI 阅读和审查工程图纸的能力（旧范式），SD-HWE-Bench 评估 AI 自主生成合规可交付工程的能力（新范式）。

- **RLVR 社区（DeepSeek-R1, GRPO）**：SD-HWE-Bench 的 L0-L4 分层验证直接服务于 RLVR 训练——秒级反馈、确定性奖励、可解释失败、分层课程学习。工程领域将成为 RLVR 研究的下一片疆土。

---

## 九、结语

> Game → Math → Code → HWE

这条路径不是预测，而是对 RLVR 方法论自然延伸的推断。每一个成功扩张的领域都满足两个条件：**约束与质量天然高相关，验证廉价且确定性。** 硬件工程完美满足这两个条件——它只是缺少把约束暴露出来的基础设施。

谁先补上基础设施缺口，谁就为整个工程 AI 领域打开了 RLVR 的飞轮。SD-HWE-Bench 是这条路上的第一步。
